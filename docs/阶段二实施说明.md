# 阶段二实施说明：查询与导出优化

目标：生效/已撤销名单改为**库内分页 + 库内筛选 + 排序**，审计日志导出**分批/限制条数**，教师端批量比对**学号分批 IN 查询**，并补充**索引**以支撑千人规模。  
每步说明：**做什么 → 怎么做 → 作用与优化点**。

---

## 步骤 1：为 Blacklist、AuditLog 增加索引（`core/models.py`）

### 做什么

- 为 Blacklist 增加组合索引：`(status, id_card)`、`(status, name)`，便于按状态 + 学号/姓名筛选与排序。
- 为 AuditLog 增加单列索引：`timestamp`，便于按时间倒序的列表与导出。

### 怎么做

- 在 `Blacklist` 上设置 `__table_args__ = (Index("ix_blacklist_status_id_card", "status", "id_card"), Index("ix_blacklist_status_name", "status", "name"))`（列名 `id_card` 与表定义一致）。
- 在 `AuditLog` 上设置 `__table_args__ = (Index("ix_audit_log_timestamp", "timestamp"),)`。
- 从 `sqlalchemy` 增加导入 `Index`。

### 作用与优化点

- **作用**：按 status 筛选、按学号/姓名排序及分页时，数据库可使用索引，减少全表扫描。
- **优化点**：审计日志按时间倒序展示与导出时，可走 `timestamp` 索引，列表与导出性能更稳定。

---

## 步骤 2：生效名单改为库内分页 + 筛选 + 排序（`views/admin_page.py`）

### 做什么

- 生效名单不再一次性 `query(...).all()` 再在内存中筛选、排序、分页；改为在数据库内完成筛选（姓名/学号/专业 LIKE）、排序（姓名/学号/专业/处分日期）、分页（`offset`/`limit`），总数用 `count()`。
- 导出「当前筛选的生效名单」时，按同一筛选与排序条件分批查询（每批 2000 条，最多 50000 条），再生成 Excel，避免单次全量加载导致 OOM 或超时。

### 怎么做

- 新增 `_effective_base_query(db, fn, fs, fm)`：`status == 1`，并按姓名/学号/专业做 `LIKE %val%` 筛选。
- 新增 `_effective_order(q, sort_key_eff, sort_asc_eff)`：按选定列 `order_by` 升序/降序。
- `_render_effective_list_section`：先取筛选与排序参数，再 `base = _effective_base_query(...)`，`total_eff = base.count()`，`ordered = _effective_order(base, ...)`，按当前页 `offset/limit` 只取一页数据；表格与分页仅使用当前页与总数。
- 导出：`_fetch_effective_export_rows` 按同一条件分批 `offset/limit` 拉取（`EXPORT_BATCH_SIZE=2000`，`EXPORT_MAX_ROWS=50000`），合并后生成 Excel；若超过 50000 条则提示「仅导出前 50000 条」。

### 作用与优化点

- **作用**：名单变大时内存占用与响应时间不再随全表线性增长，仅与当前页大小相关。
- **优化点**：筛选、排序、总数、分页均在库内完成，配合步骤 1 的索引，适合千人并发与大数据量。

---

## 步骤 3：已撤销名单改为库内分页 + 筛选 + 排序（`views/admin_page.py`）

### 做什么

- 与生效名单一致：已撤销名单改为库内筛选（`status == 0` + 姓名/学号/专业 LIKE）、库内排序与分页，总数用 `count()`；导出按同一条件分批拉取，最多 50000 条。

### 怎么做

- 新增 `_revoked_base_query(db, rn, rs, rm)`、`_revoked_order(q, sort_key_rev, sort_asc_rev)`。
- `_render_revoked_section`：与生效名单相同流程——先筛选/排序参数，再 `base.count()`、`ordered.offset/limit` 取一页，表格与分页使用当前页与总数；导出使用 `_fetch_revoked_export_rows` 分批拉取并生成 Excel。

### 作用与优化点

- **作用**：已撤销名单与生效名单行为一致，大数据量下不占满内存、不超时。
- **优化点**：同样利用 Blacklist 的 `(status, id_card)`、`(status, name)` 索引。

## 步骤 4：审计日志导出改为分批 + 限制条数（`views/admin_page.py`）

### 做什么

- 列表仍为「当前筛选下最多 500 条」展示；导出不再一次性 `q_export.all()`，改为按同一筛选条件分批查询（每批 2000，最多 50000 条），再生成 Excel，并提示「当前筛选共 N 条，最多导出 50000 条」。

### 怎么做

- 抽取 `_audit_log_export_query(db, ...)` 构建与列表同条件的导出用查询。
- `_fetch_audit_logs` 改为返回 `(logs, total_export)`：列表用 `q.limit(500).all()`，总数用 `q.count()`（或先 count 再 limit 取 500）。
- 新增 `_fetch_audit_logs_export_batched`：按 `EXPORT_BATCH_SIZE`/`EXPORT_MAX_ROWS` 分批 `offset/limit` 拉取，合并后供导出使用。
- `_render_audit_log_display` 在生成导出 Excel 时调用分批接口，并在界面上说明「共 N 条，最多导出 50000 条」。

### 作用与优化点

- **作用**：审计日志量很大时，导出不再 OOM 或长时间阻塞；列表保持轻量。
- **优化点**：配合 `ix_audit_log_timestamp` 索引，按时间倒序的批量 offset/limit 更高效。

## 步骤 5：教师端批量比对学号分批 IN 查询（`views/teacher_page.py`）

### 做什么

- 教师端批量比对时，上传 Excel 解析出的学号列表不再一次性 `student_id.in_(student_ids)`；改为按批（如每批 500）拆成多次 `in_(batch)` 查询，结果合并去重（同一学号只会在一个 batch 中出现一次，直接 extend 即可）。

### 怎么做

- 将 `student_ids` 按 `BATCH_SIZE = 500` 分片，循环 `for i in range(0, len(student_ids), BATCH_SIZE)`，每批 `batch = student_ids[i : i + BATCH_SIZE]`，执行 `query(Blacklist).filter(Blacklist.status == 1, Blacklist.student_id.in_(batch)).all()`，将结果 `matched.extend(batch_records)`。

### 作用与优化点

- **作用**：学号上万时，避免单次 IN 列表过长导致部分驱动或数据库报错或性能骤降。
- **优化点**：每批 500～1000 条 IN 查询稳定可预期，与阶段一连接池配合更好。

## 已有表建索引（表已存在时执行）

若在阶段二之前已有 `blacklist`、`audit_logs` 表，`init_db.py` 的 `create_all` 不会自动补建索引，需在对应数据库中执行：

**MySQL：**

```sql
CREATE INDEX ix_blacklist_status_id_card ON blacklist (status, id_card);
CREATE INDEX ix_blacklist_status_name ON blacklist (status, name);
CREATE INDEX ix_audit_log_timestamp ON audit_logs (timestamp);
```

**PostgreSQL：**

```sql
CREATE INDEX ix_blacklist_status_id_card ON blacklist (status, id_card);
CREATE INDEX ix_blacklist_status_name ON blacklist (status, name);
CREATE INDEX ix_audit_log_timestamp ON audit_logs (timestamp);
```

**SQLite：**

```sql
CREATE INDEX ix_blacklist_status_id_card ON blacklist (status, id_card);
CREATE INDEX ix_blacklist_status_name ON blacklist (status, name);
CREATE INDEX ix_audit_log_timestamp ON audit_logs (timestamp);
```

## 阶段二审计与修复（记录）

- **LIKE 筛选**：对姓名/学号/专业筛选输入做 `%`、`_`、`\` 转义（`_like_escape`），使用 `LIKE ... ESCAPE '\\'`，避免用户输入导致匹配过宽或行为异常。
- **审计日志查询**：列表条数与展示用 500 条改为两次独立查询（先 `count()`，再 `limit(500).all()`），避免依赖同一 Query 对象复用。
- **死代码与幻觉**：移除阶段二后已无调用的内存版筛选/排序函数（`_filter_effective_list`、`_sorted_effective_list`、`_sort_key_by_attr`、`_filter_revoked_list`），避免虚构引用、提升可读性与一致性。
