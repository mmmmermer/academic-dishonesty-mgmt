本文档基于**当前代码与文档**，从**模块功能**和**系统稳定性**两方面对系统进行评判，并给出基于行业基准的估值区间。数据来源与国标/行业标准见前文说明（GB/T 36964-2018、BSCEA 中国软件行业基准数据）。

## 二、系统稳定性评估

### 2.1 安全与合规

| 维度 | 情况 | 说明 |
|------|------|------|
| 密码存储 | ✅ | BCrypt，无明文；长度超限拒绝，防异常 |
| 登录防护 | ✅ | 失败次数限制 + 冷却时间（可配置） |
| 会话安全 | ✅ | 服务端会话存储，超时失效；sid 恢复不暴露敏感信息 |
| 审计追溯 | ✅ | 登录、查询、导入、增删、恢复、备份等均记入 audit_logs，只增不减 |
| 输入与注入 | ✅ | 名单筛选参数化 + LIKE 转义（_like_escape）；水印 html.escape |
| 敏感信息 | ✅ | 日志不记密码；DATABASE_URL 不写日志不展示；恢复仅 SQLite 且二次确认 |

结论：**安全设计完整**，满足高校内部系统对身份、审计与数据保护的要求。

### 2.2 数据与运维可靠性

| 维度 | 情况 | 说明 |
|------|------|------|
| 备份 | ✅ | SQLite 下自动备份 + 手动下载；MySQL/PG 文档与示例脚本（mysqldump/pg_dump） |
| 恢复 | ✅ | SQLite 下上传覆盖，恢复后双清 cache 并提示刷新/重启 |
| 数据库可选 | ✅ | SQLite / MySQL / PostgreSQL，连接池与迁移脚本就绪 |
| 索引与性能 | ✅ | Blacklist (status, id_card)、(status, name)；AuditLog (timestamp)；大表库内分页、导出分批 |
| 缓存一致性 | ✅ | 恢复后 st.cache_data.clear() + st.cache_resource.clear()，避免脏缓存 |

结论：**数据与运维**具备备份、恢复、多库与性能优化，适合正式部署。

### 2.3 可扩展与高可用

| 维度 | 情况 | 说明 |
|------|------|------|
| 多实例部署 | ✅ | Nginx 示例、多进程启动脚本、会话粘滞；多实例下 SQLite 检查与提示 |
| 会话多进程 | ✅ | session_store 文件锁（读/写锁），多实例共享 sessions.json 时安全 |
| 规模演进 | ✅ | 文档明确：换库+多实例+分页/导出优化即可支撑约千人，无需推倒重做 |

结论：**扩展路径清晰**，稳定性可随规模演进。

### 2.4 错误处理与可维护性

| 维度 | 情况 | 说明 |
|------|------|------|
| 用户可见错误 | ✅ | st.error 等中文提示，不直接暴露 Traceback |
| 耗时操作反馈 | ✅ | st.spinner 等，文档有要求 |
| 日志 | ✅ | 登录成功/失败、导入、删除/恢复、备份/恢复等关键路径打 logger；LOG_SUBDIR/LOG_FILENAME 可配置 |
| 配置集中 | ✅ | core/config.py 统一常量与环境变量，便于运维与 i18n |

结论：**错误处理与可维护性**达到可交付水平。

### 2.5 自动化测试

| 维度 | 情况 | 说明 |
|------|------|------|
| 单元/集成测试 | ⚠️ 缺失 | README 提及 `pytest tests/`，但项目中无 tests 目录或 test_*.py；无自动化回归保障 |

结论：**稳定性在“测试”维度有短板**；若用于对外报价或长期维护，建议补充核心流程的自动化测试。

### 2.6 稳定性综合

- **强项**：认证与会话、审计、备份与恢复、多库与多实例支持、索引与分批、日志与配置。
- **弱项**：无自动化测试，回归依赖人工；对长期维护与招投标中的“质量保证”条款会稍显不足。

## 三、按模块与稳定性折算的规模与价值

### 3.1 功能规模（供套用基准数据）

按**功能点思路**（与 GB/T 36964、BSCEA 体系兼容）对当前实现做**粗略计数**，仅作估值参考，非正式功能点评估：

| 类别 | 内容 | 约估功能点（UFP 量级） |
|------|------|------------------------|
| 登录与会话 | 登录、超时、恢复、水印 | 15～25 |
| 教师端 | 单条查询、批量比对、结果导出 | 25～40 |
| 仪表盘 | 统计、图表、近期变动 | 20～35 |
| 名单管理 | 导入、增删改、生效/撤销列表、筛选排序分页、比对 | 45～70 |
| 系统维护 | 审计列表与导出、备份下载、恢复 | 25～40 |
| 用户管理 | 列表、新增、密码重置、启用禁用 | 20～30 |
| **合计** | | **约 150～240** |

说明：上述为经验化区间；正式报价建议按国标或 BSCEA 实施规程做一次规范功能点计数。

### 3.2 稳定性对价值的影响（系数）

在“功能点 × 单价”基础上，用**稳定性系数**做微调（不改变数量级，仅区分档次）：

| 维度 | 权重 | 评分（1～5） | 说明 |
|------|------|--------------|------|
| 安全与审计 | 高 | 5 | BCrypt、会话、审计、注入防护到位 |
| 数据与备份恢复 | 高 | 5 | 备份/恢复、多库、迁移、缓存一致性 |
| 性能与扩展 | 中 | 4 | 索引、分页、分批、多实例；单实例 SQLite 有并发上限 |
| 可维护与运维 | 中 | 4 | 日志、配置、部署文档与示例齐全 |
| 测试与回归 | 中 | 2 | 无自动化测试 |
| **综合** | | **约 4.0/5** | 稳定性整体中上，缺测试拉低约 0.5～1 档 |

建议：**稳定性系数取 0.90～0.98**（即在不考虑测试的前提下，按接近“完整交付物”计价；若后续补足测试，可视为 1.0）。

### 3.3 估值区间（基于 BSCEA 基准）

- **基准**：BSCEA《中国软件行业基准数据》中 2025 年北京功能点单价约 **1243.52 元/功能点**（公开报道值；具体以官网下载的 Excel 为准）。
- **规模**：按上表取中位约 **195 功能点**；若保守取 **150**、上限取 **240**。
- **未调整开发成本**（仅功能点 × 单价）：
  - 150 × 1243.52 ≈ **18.7 万**
  - 195 × 1243.52 ≈ **24.2 万**
  - 240 × 1243.52 ≈ **29.8 万**
- **乘以稳定性系数 0.90～0.98**（体现无自动化测试的折价）：
  - 下限：18.7 × 0.90 ≈ **16.8 万**
  - 中位：24.2 × 0.95 ≈ **23.0 万**
  - 上限：29.8 × 0.98 ≈ **29.2 万**

以上为**开发成本/造价**量级。若作为**对外报价（含利润、实施与培训）**，常见在成本基础上上浮 10%～30%，即约 **18 万～38 万** 的报价区间；若仅做“产品价值评估”而非投标价，可直接用 **17 万～30 万** 作为系统价值的参考区间。

## 四、结论与金额区间

- **按模块功能**：需求模块全部实现，且具备库内分页、索引、分批导出、多库与多实例、迁移与部署文档等，**功能完整度与可交付性高**，可支撑 **150～240 功能点** 量级的估算。
- **按系统稳定性**：安全、审计、备份恢复、多库与扩展、日志与配置均到位，**稳定性中上（约 4.0/5）**；主要不足为**无自动化测试**，用系数 0.90～0.98 做小幅折价。
- **综合估值**（基于当前模块功能与稳定性，并套用 BSCEA 功能点单价与稳定性系数）：
  - **开发成本/造价**：约 **17 万～30 万**（人民币）。
  - **对外报价参考**：约 **18 万～38 万**（含实施与合理利润）。

若后续**补充核心流程的自动化测试**，稳定性系数可视为 1.0，上述区间可整体上移约 2%～10%，并更利于在招投标或审计中作为“质量保障”依据。

## 五、按电商与接单市场的价格评判

以下按**电商（淘宝/闲鱼等）与接单市场（猪八戒/工作室/个人接单）**的常见报价习惯，对当前系统做价格区间评判，与上文「国标/基准」估值并列，供不同销售场景选用。

### 5.1 市场上常见价位（参考）

| 类型 | 典型价格 | 说明 |
|------|----------|------|
| 毕设/源码成品 | 几十～几百元 | 淘宝/闲鱼常见：单表 CRUD、无审计/备份、无部署文档，多为学习用 |
| 套壳+简单模块 | 约 2000～5000 元 | 通用后台 + 改表名/菜单，名单增删改查；多数无完整审计、无多库/迁移 |
| 小型定制（展示/简单业务） | 5000～2 万 | 接单市场：简单企业站、表单、基础后台；周期短、功能单一 |
| 小型定制（多模块/可运维） | 1 万～3 万 | 带权限、日志、备份或部署说明；业务完整、可上线使用 |
| 学校/教育定制（多端、多模块） | 十几万～几十万 | 猪八戒等：智慧校园、多端、复杂业务；与本系统「单校名单+查询」不同档 |

### 5.2 本系统在电商/接单市场中的定位

- **功能**：单校、名单管理 + 教师查询 + 审计 + 备份恢复 + 双角色 + 多库/多实例支持，**高于「套壳+改表」**，**低于「智慧校园/多端平台」**。
- **交付物**：完整源码 + 需求与架构文档 + 部署与迁移说明 + 可运行、可扩展，**达到「小型定制项目交付」水平**。

### 5.3 按电商/接单市场给出的报价区间

| 销售方式 | 建议报价区间（人民币） | 说明 |
|----------|------------------------|------|
| **仅卖源码（不包部署、不包改）** | 3000～8000 | 比毕设档贵、比 2000 套壳功能完整，适合「能自己部署」的买家 |
| **源码 + 部署 + 简单培训** | 5000～1.5 万 | 包环境搭建、一次部署、使用说明，适合学校/部门自用 |
| **按「接单定制项目」交付** | 1 万～3 万 | 视为已按需求做完的定制项目，含文档与运维能力，对标接单市场「小型完整项目」 |
| **多校复制（按校授权）** | 单校 3000～8000 | 同一套产品多校售卖时，单校授权价可落在该区间，总价看校数 |

### 5.4 与「2000 左右」的对比

- 2000 档多为：通用架构 + 名单/学生等模块的**改名与简单 CRUD**，无完整审计、无备份恢复、无多库与迁移、无部署与演进文档。
- 本系统具备**完整业务规则、审计、安全、备份与恢复、文档与扩展路径**，在电商/接单市场更接近 **5000～3 万** 的「小型完整项目」档，而不是 2000 的「套壳」档；具体取 5000、1 万还是 2 万，可按渠道、是否包部署与培训、是否多校复制来定。
