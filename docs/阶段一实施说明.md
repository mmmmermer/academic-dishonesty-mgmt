# 阶段一实施说明：数据库与连接层改造

目标：支持 MySQL/PostgreSQL，增加连接池，为千人并发与多实例部署打基础。  
每步说明：**做什么 → 怎么做 → 作用与优化点**。

---

## 步骤 1：改造 `core/database.py`（支持环境变量 URL + 连接池）

### 做什么

- 从环境变量读取 `DATABASE_URL`；未设置时仍使用当前 SQLite 路径，保证本地开发零配置。
- 根据 URL 类型选择引擎参数：SQLite 保留 `check_same_thread=False`；MySQL/PostgreSQL 使用连接池（`pool_size`、`pool_pre_ping`、可选 `pool_recycle`），且不再使用 SQLite 专有参数。
- 对外暴露一个布尔量（如 `IS_SQLITE`），供备份/下载等逻辑判断是否操作本地文件。

### 怎么做

- 使用 `os.environ.get("DATABASE_URL")`，为空或未设置时 `DATABASE_URL = f"sqlite:///{os.path.join(DATABASE_DIR, 'database.db')}"`。
- 若 `"sqlite" in DATABASE_URL`：`create_engine(..., connect_args={"check_same_thread": False})`，不设 pool_size（SQLite 默认单连接即可）。
- 否则：`create_engine(..., pool_size=10, max_overflow=5, pool_pre_ping=True, pool_recycle=3600)`，不传 `connect_args`。
- 定义 `IS_SQLITE = "sqlite" in (DATABASE_URL or "")`，供 `core.utils` 等使用。

### 作用与优化点

- **作用**：同一套代码可通过环境变量切换 SQLite（本地/测试）与 MySQL/PostgreSQL（生产），为后续多实例共享同一数据库做准备。
- **优化点**：连接池避免每次请求新建连接，降低连接建立/断开开销；`pool_pre_ping` 减少因数据库重启或网络抖动导致的“死连接”；`pool_recycle` 避免长时间连接被服务端关闭导致的中断。

## 步骤 2：更新 `requirements.txt`（增加可选驱动）

### 做什么

- 在依赖中增加 MySQL、PostgreSQL 的 Python 驱动，便于生产环境安装后即可使用对应数据库。

### 怎么做

- 增加 `pymysql`（MySQL，纯 Python，易安装）、`psycopg2-binary`（PostgreSQL 官方推荐二进制包）。若希望与 SQLite 完全兼容、生产仅用其一，可注明“按需安装其一”。

### 作用与优化点

- **作用**：安装后可通过 `DATABASE_URL` 使用 MySQL（`mysql+pymysql://...`）或 PostgreSQL（`postgresql+psycopg2://...`），无需改业务代码。
- **优化点**：`psycopg2-binary` 自带 libpq，避免单独编译；`pymysql` 无需 C 扩展，部署简单。

---

## 步骤 3：兼容 `core/utils.py` 中的备份与下载

### 做什么

- 当当前数据库**不是** SQLite 时，“下载 database 文件”和“启动时自动备份到本地文件”无法再以复制 `.db` 文件的方式实现，需做兼容。

### 怎么做

- 在 `utils.py` 中从 `core.database` 导入 `IS_SQLITE`（或等价标志）。
- `get_db_file_bytes()`：若 `not IS_SQLITE`，抛出明确异常（如 `NotImplementedError` 或自定义提示），提示生产环境使用 mysqldump/pg_dump 等工具备份。
- `auto_backup()`：若 `not IS_SQLITE`，直接 return（不写文件），可选打日志说明“已使用外部数据库，请通过 DB 工具备份”。

### 作用与优化点

- **作用**：避免在 MySQL/PostgreSQL 下误用“下载 .db”或“复制 .db”逻辑，防止运行时错误或误导性行为。
- **优化点**：保留 SQLite 下的现有行为；生产环境备份方式在文档中说明（mysqldump/pg_dump 或云厂商快照），后续可在阶段四再考虑封装调用。

## 步骤 4：数据迁移脚本与使用方式

### 做什么

- 提供“从现有 SQLite 迁移到当前配置的数据库（多为 MySQL/PostgreSQL）”的脚本，使已有数据可一次性导入新库。
- 不改变 `init_db.py` 行为：仍负责“建表 + 默认管理员”；迁移脚本负责“从 SQLite 读数据并写入当前引擎”。

### 怎么做

- 新增脚本（如 `scripts/migrate_sqlite_to_external.py` 或项目根目录 `migrate_sqlite_to_external.py`）：
  - 使用固定路径或环境变量指向现有 `database.db`，用 SQLite 引擎只读打开。
  - 当前运行环境的 `DATABASE_URL` 作为目标（即 `core.database` 的 engine），若为 SQLite 则提示“目标应为 MySQL/PostgreSQL”。
  - 在目标上执行 `Base.metadata.create_all(bind=target_engine)`（若表已存在则跳过）。
  - 按表顺序（如 User → Blacklist → AuditLog）从 SQLite 全表读取再批量写入目标（每批若干条 commit），避免长事务与内存峰值。
- 在 `docs/阶段一实施说明.md` 或 README 中写明：先设置好目标库的 `DATABASE_URL`，再执行 `python migrate_sqlite_to_external.py`；若目标库为空，也可直接 `python init_db.py` 建表并只创建默认管理员。

### 作用与优化点

- **作用**：已有 SQLite 数据的环境可平滑迁到 MySQL/PostgreSQL，无需手工导表。
- **优化点**：批量写入 + 分批 commit 控制事务长度与内存；先建表再导数据，避免依赖顺序问题；脚本幂等（可多次执行，已存在数据时需考虑跳过或覆盖策略，建议“目标无数据时使用”）。

## 阶段一完成后的状态

- 应用通过 `DATABASE_URL` 可连 SQLite（默认）或 MySQL/PostgreSQL。
- 使用 MySQL/PostgreSQL 时启用连接池与健康检查，适合多请求并发。
- 备份/下载逻辑在非 SQLite 下不再依赖本地 .db 文件，由文档约定外部备份方式。
- 提供从 SQLite 到外部库的一次性迁移脚本，便于现有数据迁到生产库。
- 业务代码（views、auth、utils 中与 DB 无关部分）无需修改；仅数据库连接层与备份相关逻辑有改动。

## 运行前检查清单（使用 MySQL/PostgreSQL 时）

1. **安装驱动**：`pip install pymysql`（MySQL）或 `pip install psycopg2-binary`（PostgreSQL）。
2. **创建数据库**：在 MySQL/PostgreSQL 中先建好空库（如 `hustsystem`），并记下连接信息。
3. **设置环境变量**（示例）：
   - MySQL：`DATABASE_URL=mysql+pymysql://用户:密码@主机:3306/库名?charset=utf8mb4`
   - PostgreSQL：`DATABASE_URL=postgresql+psycopg2://用户:密码@主机:5432/库名`
4. **建表与初始数据**：
   - 若目标库为空：执行 `python init_db.py`（会建表并创建默认管理员 admin/123456）。
   - 若要从现有 SQLite 迁数据：执行 `python migrate_sqlite_to_external.py`（会建表并复制 User/Blacklist/AuditLog）。
5. **启动应用**：`python -m streamlit run app.py`（确保同一环境中 DATABASE_URL 已设置）。

---

## 生产备份建议（非 SQLite）

- **MySQL**：使用 `mysqldump -u用户 -p 库名 > backup.sql` 或云厂商自动快照。
- **PostgreSQL**：使用 `pg_dump -U用户 库名 -f backup.sql` 或云厂商快照。
- 管理后台中的「下载当前数据库」在 MySQL/PostgreSQL 下会提示使用上述工具，不会提供 .db 文件下载。
