# 华中科技大学学术道德监督管理系统 — 高安全性局域网部署与硬件指南

> **文档版本**：v1.0  
> **适用场景**：高保密性、安全性要求极高，仅限同一办公室（或特定科室）局域网人员内部使用，杜绝一切外网访问与云端泄露风险。

---

## 一、部署架构核心理念：内网“铁桶”模型 (Intranet Vault)

在极高保密与极低并发（几人~十几人同频使用）的特定场景下，**系统的防御痛点不是“扛压防崩”，而是“防内鬼抓包、防物理失窃、收缩网络攻击面”**。

这套高安全性部署架构必须恪守以下四条铁律：

1. **绝对物理/逻辑网络隔离**：服务器禁止绑定任何公网入站或出站 IP，切断所有跨公网数据流。
2. **反向代理 TLS 加密套壳**：在即使全可信的办公室内网，也严禁直接以 `http://ip:8501` 明文裸奔，必须由 Nginx 提供 HTTPS 加密（自签名证书即可），以阻断局域网内的 Wireshark 等嗅探软件还原数据。
3. **保留并优化单文件数据库**：在低并发受限机器上，**内置纯原生文件型数据库 SQLite 的安全性远大于 MySQL**。因为 SQLite 没有默认的 `3306` 网络端口，即使同一大楼的办公网段有人中了勒索病毒，扫描器也找不到数据库端口进行爆破脱库！
4. **底层磁盘 LUKS 物理加密**：对于数以千计含有极敏感隐私的 PDF 违纪通报，最致命的泄密是“带走硬盘”。因此，宿主机的系统盘/数据盘必须上 Linux LUKS (Linux Unified Key Setup) 硬件级加密。

---

## 二、 软硬件选型与系统基线支撑

通过扫描 `requirements.txt` 和本源码，系统为极其轻量的 I/O 驱动型应用。不需要昂贵的集群服务器。

**系统运行底线需求：**
- **CPU**：普通双核（如 i3）即存在极度性能溢出（日常负载 <10%）
- **内存**：≥ 8GB（操作系统占带 2G，Python 环境占带 1G，数据缓冲占 2G，绰绰有余）
- **硬盘**：SSD 固态至少 128GB（系统 + 黑名单占用 < 1GB，主要是未来几十万份 PDF 需要空间）

### 🏆 官方推荐硬件方案评测

#### 方案 A（性价比首选）：零刻 SER6 Pro 迷你主机 等类似 x86 微型主机
* **定位**：放置于办公室文件柜中，体积仅手掌大，功耗极低。
* **参考配置**：AMD Ryzen 7 6800H / 16G DDR5 / 500G NVMe SSD (约 ¥1500)
* **性能溢出极高**：跑这套系统毫无压力，即使未来增配多路数据库或大模型检索也足以应付 10 年以上。

#### 方案 B（数据容灾首选）：群晖 DS723+ 等企业级 NAS
* **定位**：重视系统数据异地容灾与 PDF 原件硬盘物理损毁防护。
* **参考配置**：AMD R1600 / 必须升级至 16G 内存 / 双 4TB NAS 红盘组 RAID1 (约 ¥4500-5000)
* **最大优势**：两块硬盘实时互相镜像。即使其中一块硬盘使用 5 年后突然坏道、死机，系统在不停机的情况下换上一块新硬盘就能自动修复所有上万份 PDF 数据。

## 三、 高安全生产环境部署实施清单（Steps）

无论选择哪套硬件，请让基础设施配置人员严格对照此清单进行装配：

### 第一步：操作系统环境夯实
1. **安装 OS**：强烈建议安装 **Ubuntu 22.04 LTS (Server 版无图形界面)**，去除所有无关的多媒体及桌面组件。
2. **LUKS 全盘加密**：在安装 OS 格式化磁盘时，务必勾选 `Encrypt the new Ubuntu installation for security (LVM with LUKS)`。这样开机时必须连接显示器键盘输入加密密码才能进系统，物理拔盘数据将彻底沦为一堆乱码废铁。
3. **账户隔离**：不要使用 `root` 运行该系统。新建一个无 Bash 登录权限的受限系统用户：`sudo useradd -r -s /bin/false hust_app`。

### 第二步：数据库策略收紧（当前代码已就绪）
1. 我们已经在 `core/database.py` 中为您**开启了 SQLite 的 `WAL` （Write-Ahead Logging）防锁高并发模式**。几位老师同时录入比对不会产生死锁报错。
2. **锁定权限**：将 `/app/database.db` 文件的所有权仅授予给 `hust_app` 服务账户（`chmod 600 database.db`），杜绝全系统中其他任何人 copy 走。

### 第三步：Streamlit 后台守护服务搭建
使用 Systemd 托管代码（崩溃即时重启）。

在 `/etc/systemd/system/hustsystem.service` 写入配置：
```ini
[Unit]
Description=HUST Academic Integrity System
After=network.target

[Service]
User=hust_app
WorkingDirectory=/opt/hustsystem
ExecStart=/opt/hustsystem/.venv/bin/streamlit run app.py --server.port 8501 --server.address 127.0.0.1
Restart=always

[Install]
WantedBy=multi-user.target
```
> **注意安全眼**：`--server.address 127.0.0.1` 意味着该 Streamlit 进程**彻底切断了对外围网线的监听**，局域网内的任何其他电脑都无法直接打入。我们只交给下方的 Nginx 转发！

### 第四步：Nginx 反向代理与 HTTPS 加密（防御网线级窃听核心）
在电脑上安装 Nginx 并自签发 SSL 加密证书。

**Nginx 核心配置段 `/etc/nginx/sites-available/hustsystem` 示例：**
```nginx
server {
    listen 443 ssl;
    server_name 192.168.10.x; # 宿主机的内网IP

    # 自签名 HTTPS 证书（防御抓包）
    ssl_certificate /etc/ssl/certs/hustsystem_selfsigned.crt;
    ssl_certificate_key /etc/ssl/private/hustsystem_selfsigned.key;

    # 办公室网段 IP 白名单（防御内网横向扫描）
    allow 192.168.10.0/24;  # 请替换为您科室内的实际网段
    deny all;               # 非本办公室的一律拉闸拒绝

    location / {
        # 将流量安全解密后，本地传递给后后方锁死的 Streamlit
        proxy_pass http://127.0.0.1:8501;
        
        # Streamlit 必须配置好 WebSocket 的 Upgrade 透传机制
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_addrs;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

> ⚠️ **【重要网络避坑指南】校园网 IP 白名单陷阱**  
> 在真实的大学校园网中，简单设置 `allow 192.168.10.0/24;` 极大概率会遇到以下问题，请提前与网信办确认或采取备用方案：
> 1. **“大子网”陷阱**：学校可能是按整栋楼（甚至几个学院）划分的大网段（如 `10.x.0.0/16`）。如果白名单放得太宽，意味着整栋楼的几千个学生都能访问到登录页，失去了“仅办公室可见”的隔离初衷。
> 2. **动态 IP 变动**：老师们的电脑通常是 DHCP 自动获取 IP，今天连有线网是内网 IP，明天连校园 WiFi（如 EDUROAM）就变成了另一个网段的 IP，导致老师频繁被阻挡在外。
> 3. **NAT 及 VPN 转化**：如果老师下班后用学校 VPN 在家连，查到的源 IP 将全变成 VPN 网关的 IP。
> 
> **🎯 针对校园网的根本解决方案：**
> * **首选方案（硬件绑定）**：请学校网管中心（网络中心）在后台为本办公室几位老师的电脑网卡 **MAC 地址绑定静态 IP**（或划分专属 VLAN 小网段），然后 Nginx 中只 `allow` 这仅有的几个固定 IP。
> * **次选方案（Nginx 密码墙）**：如果学校无法分配静态 IP，必须放开白名单网段，则**必须在 Nginx 层叠加二次通关密码（HTTP Basic Authentication）**。
>   * 在 Nginx location 中加入：
>     `auth_basic "Restricted Area";`
>     `auth_basic_user_file /etc/nginx/.htpasswd;`
>   * 这样一来，即使别的学院的人扫描到了该网址，也必须先输入一道“通关口令”才能看到 Streamlit 的系统登录页，有效防止了系统的被动扫描。

### 第五步：U 盘离线冷备机制落地
现在我们在代码的备份逻辑中配置了报错在 UI 层面发出高亮警报的功能（见更新日志 `core/utils.py`）。
考虑到硬件仍可能存在“带病双挂”的风险，建议指定一名管理员，每周定期将 `/opt/hustsystem/backups/` 目录下带有最新时间戳的 `database_xxxx.db` 拷贝至本人的最高机密加密 U 盘中，放入部门保险柜。由于有了先前的优化，旧的高频更新的废弃 PDF 已被系统自动清理（孤儿 PDF 垃圾回收机制），此备操作份干净且迅速。
