# 千人规模评估与演进建议

本文档基于对当前系统全部代码与文档的阅读，回答：**为支撑约一千人同时使用、系统流畅不崩溃并上线服务器，现有文件结构/前后端架构/技术栈有哪些不足？是否建议删除全部代码、严格前后端分离并重新设计开发？**

---

## 一、当前系统概览（需求与功能总结）

### 1.1 业务定位

- **产品**：校园学术失信人员管理系统（名单管理、公示、查询）。
- **用户**：管理员（全功能）、教师（只读查询）。
- **核心流程**：Excel 导入名单 → 学号清洗 → SQLite 存储 → 单条/批量查询与比对、名单维护、审计、备份、用户管理。

### 1.2 功能清单（现有实现）

| 模块 | 功能点 | 实现要点 |
|------|--------|----------|
| **认证与会话** | 工号+密码登录、刷新保持登录、会话超时、登出 | BCrypt 校验；`sessions.json` + URL `sid` 恢复；`st.session_state` |
| **教师端** | 单条查询（姓名/学号，多行）、批量比对（Excel 上传）、个人操作记录 | 仅查 status=1；学号清洗；分页展示命中/未命中；Excel 下载 |
| **管理员端** | 仪表盘（总数/生效/撤销、专业饼图、年份柱状图、近期变动） | Plotly；全量/筛选后内存分页 |
| **管理员端** | 名单管理：批量导入、单条新增、生效列表（筛选/排序/分页）、软删除、已撤销列表与恢复、编辑 | Excel 必需列校验；学号唯一；部分列表为**全量加载后内存分页** |
| **管理员端** | 系统维护：审计日志（列表+导出）、手动备份下载、数据库恢复上传 | 最近 500 条；全量导出；`get_db_file_bytes` / 恢复逻辑 |
| **管理员端** | 用户管理：列表、新增、密码重置、启用/禁用 | 不可禁用当前用户 |
| **全局** | 水印（姓名+工号）、错误中文提示、自动备份（启动时） | 配置在 `core/config.py` |

### 1.3 技术栈与文件结构（现状）

- **技术栈**：Streamlit（UI + 路由 + 状态）、SQLite、SQLAlchemy、Pandas/OpenPyxl/xlrd、BCrypt、Plotly。
- **文件结构**：`app.py`（入口+会话+侧栏）+ `core/`（config、database、models、auth、utils、session_store）+ `views/`（login、teacher_page、admin_page）+ `init_db.py`。
- **架构**：前后端**未分离**——页面与业务逻辑均在 Python 中，Streamlit 服务既是「后端」也是「前端」渲染端；无独立 REST API、无独立前端工程。

## 二、面向「一千人同时使用、流畅不崩溃、上线服务器」的不足

### 2.1 数据库与存储

| 不足 | 说明 |
|------|------|
| **SQLite 并发** | 写串行，多用户同时写（如多人导入/修改）会锁等待；一千人并发读也会加重单文件 I/O。 |
| **无独立 DB 服务** | 单文件与进程同机，无法按数据库单独扩缩容、备份策略与当前架构强绑定。 |

### 2.2 数据访问与性能

| 不足 | 说明 |
|------|------|
| **名单列表全量加载** | 生效/已撤销名单先 `query(...).all()` 再在内存筛选、排序、分页，名单变大时内存与响应时间线性上升。 |
| **审计日志全量导出** | 导出时一次性 `.all()`，日志量大时易 OOM 或超时。 |
| **批量比对大 IN** | 教师端一次 `student_id.in_(student_ids)`，学号上万时部分驱动/库有长度或性能问题。 |
| **索引不完整** | AuditLog 无索引；Blacklist 仅部分索引，按 status+筛选+排序的分页查询未完全优化。 |

### 2.3 部署与扩展

| 不足 | 说明 |
|------|------|
| **单进程单实例** | 所有用户共用一个 Streamlit 进程，一千人在线时单机 CPU/内存成为瓶颈。 |
| **会话绑在进程内存** | `st.session_state` 不跨进程，若将来多实例部署必须配合「会话粘滞」或引入 Redis 等会话存储。 |
| **无横向扩展** | 当前架构无法通过「加机器/加进程」自然水平扩展，需配合换库+多实例方案。 |

### 2.4 前后端与架构

| 不足 | 说明 |
|------|------|
| **前后端一体** | UI 与业务都在 Streamlit 内，无法单独对前端做 CDN、多端适配或替换技术栈。 |
| **无标准 API** | 没有 REST/OpenAPI，第三方系统或未来移动端无法直接复用接口。 |
| **文件结构** | 模块划分清晰（core/views），但「视图」层同时承担展示与较多业务逻辑，千人规模下若要做细粒度缓存、限流、审计，会希望有明确的服务层与 API 层。 |

## 三、是否建议「删除全部代码 + 严格前后端分离 + 重新设计开发」？

### 结论：**不建议**为「一千人同时使用、流畅不崩溃、上线服务器」这一目标而推倒重来。

理由简述：

1. **一千人规模在现有架构上演进即可达到**  
   本仓库 `docs/系统需求与架构总文档.md` 与各阶段实施说明已说明：  
   - **换 MySQL/PostgreSQL**：改连接与迁移，业务代码几乎不动，即可获得高并发写与更大数据量能力。  
   - **多实例 + 会话粘滞**：前面加 Nginx（或负载均衡），多开若干 Streamlit 进程，同一用户固定到同一实例，**无需改 session 逻辑**，即可水平扩展。  
   - **库内分页 + 索引 + 流式导出**：在现有 views/core 上改查询与导出方式，即可避免大表全量加载和 OOM。  

2. **推倒重来的成本与风险很大**  
   - 严格前后端分离（例如 React/Vue + FastAPI/Django）：需重写所有页面、设计并实现整套 API、重新做认证/会话/权限、再对接前端，工作量大、周期长、易引入新 bug。  
   - 仅以「支撑一千人、稳定上线」为目标，**不需要**用前后端分离来换架构；换库 + 多实例 + 查询与导出优化即可满足。

3. **何时才值得考虑「严格前后端分离 + 重做」**  
   - 需要**独立移动端 App** 或**多终端统一 API** 时。  
   - 需要**公网门户、多系统对接、开放 API** 时。  
   - 产品形态或交互有**根本性变化**（如强交互、复杂工作流、实时协作）且 Streamlit 无法满足时。  

在上述情况出现之前，更稳妥的路线是：**在现有代码与架构上演进，先达到千人规模与上线目标，再视业务发展决定是否做前后端分离与重做。**

## 四、推荐演进路线（不删代码、不强制前后端分离）

目标：**约一千人同时使用、系统流畅不崩溃、可上线服务器**。

### 阶段 1：数据库与连接（必做）

- 将 **SQLite 改为 MySQL 或 PostgreSQL**：  
  - 修改 `core/database.py`（从环境变量读 `DATABASE_URL`，使用 pymysql/psycopg2）。  
  - 建库、建表（与现有 `models.py` 一致），做一次数据迁移。  
- 加**连接池**（如 `pool_size=10`, `pool_pre_ping=True`），便于多请求并发访问。

### 阶段 2：查询与导出优化（必做）

- **生效/已撤销名单**：改为**库内分页+库内筛选+排序**（`filter` + `order_by` + `offset/limit`），总数用 `count()` 或带缓存的计数。  
- **审计日志**：列表保持分页；导出改为**流式**（`yield_per` 或分批 offset）或限制最大条数并提示。  
- **教师端批量比对**：学号列表按批（如每批 500～1000）拆成多次 `in_(batch)` 查询，结果合并。  
- **索引**：Blacklist 的 `(status, student_id)`、`(status, name)` 等；AuditLog 的 `(timestamp DESC)` 及按需的筛选组合索引。

### 阶段 3：多实例部署与会话（必做）

- 使用 **Nginx（或云负载均衡）** 做反向代理，后端起**多个** Streamlit 进程（多端口或多机）。  
- 配置**会话粘滞**（如 `hash $cookie_...` 或 `ip_hash`），保证同一用户请求固定到同一实例。  
- 多实例下**必须使用 MySQL/PostgreSQL**，避免多进程写 SQLite 锁库。

### 阶段 4：运维与体验（建议）

- **备份**：数据库改用 mysqldump/pg_dump 或云快照；应用层保留「下载备份」功能时可改为调用 DB 备份接口或文件。  
- **监控与日志**：保留并规范 `logs/app.log`；必要时对关键接口做简单耗时/错误监控。  
- **可选**：仪表盘统计、名单总数等用 `@st.cache_data` 短时缓存，减轻 DB 压力。

上述阶段均可在**不删除现有业务代码、不引入独立前端项目**的前提下完成；视图层（views）仅做「调用方式与数据量」上的调整（如改为传分页参数、接库内分页结果），无需重写全部交互。

## 五、若未来要做「严格前后端分离」可如何衔接

若在达到千人规模并稳定运行后，仍决定做前后端分离（例如要上移动端或开放 API），建议：

1. **先沉淀 API 与领域逻辑**  
   - 在现有工程内抽一层「服务/用例」（如 `services/` 或 `api/`），把当前 views 中的核心逻辑封装成可被 HTTP 调用的函数；Streamlit 视图改为调用这些函数。  
   - 再为这些逻辑提供 REST API（如 FastAPI 路由），前端（含未来独立前端）只调 API。  

2. **前端逐步替换**  
   - 新页面或新端用新前端技术栈开发，对接已有 API；旧 Streamlit 页面可保留一段时间，按模块逐步下线。  

这样无需「先删光再重写」，而是**在现有系统上演进**，风险与成本更可控。

## 六、总结表

| 问题 | 结论 |
|------|------|
| 当前文件结构/前后端/技术栈对一千人并发的不足？ | 数据库（SQLite/单文件）、全量加载与全量导出、单实例、无库内分页与索引、会话不跨进程。 |
| 是否建议删除全部代码并严格前后端分离后重做？ | **不建议**。仅就「一千人、流畅、上线」而言，在现有架构上演进（换库 + 多实例 + 分页与导出优化 + 索引）即可。 |
| 建议做法？ | 按本文「四、推荐演进路线」分阶段实施；若未来有移动端/开放 API 等需求，再在现有基础上抽 API 层并逐步做前后端分离。 |
