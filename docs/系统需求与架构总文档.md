# 校园学术失信人员管理系统 — 系统需求与架构总文档

本文档为**需求、架构与运行**的单一总览，与当前代码一致。实施细节见 `docs/阶段一～四实施说明.md` 与 `deploy/README.md`。

---

## 一、系统概述

**系统名称**：校园学术失信人员管理系统（Academic Dishonesty Management System）

**业务定位**：
- 高校内部 Web 系统，用于记录、公示和查询学术失信人员名单。
- **核心逻辑**：Excel 导入名单库 → 自动清洗（学号等）→ 数据库存储 → 前端公示与比对。
- **身份标识**：名单以**学号**为唯一标识存储与比对（实现中表列名为 `id_card` 兼容旧库）。
- **部署环境**：学校办公室局域网（Windows/Linux Server）。
- **目标**：稳定、可靠、操作简单。

**核心业务流程**：
```
Excel 导入名单 → 数据清洗（学号标准化）→ 数据库存储 → 前端公示与比对查询
```

**技术特点**：
- **轻量级部署**：默认 SQLite 单文件；可选 MySQL/PostgreSQL 支撑多实例与千人规模。
- **简单易用**：Streamlit 构建 Web 界面，`st.session_state` 管理状态。
- **安全可靠**：BCrypt 密码加密、会话管理、审计日志、自动备份（SQLite 下）。
- **角色权限**：教师（只读查询）、管理员（全功能管理）。

## 四、数据模型

### 4.1 表结构（与 `core/models.py` 一致）

**users（用户表）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer | 主键，自增 |
| username | String(64) | 登录账号（工号），唯一 |
| password_hash | String(128) | BCrypt 加密 |
| full_name | String(64) | 真实姓名（水印显示） |
| role | String(16) | 'admin' / 'teacher' |
| is_active | Boolean | 账号状态（默认 True） |

**blacklist（失信名单表）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer | 主键，自增 |
| name | String(64) | 姓名（索引） |
| student_id | String(32) | 学号唯一（库列名 id_card 兼容旧库） |
| major | String(128) | 所学专业 |
| reason | Text | 失信/违规原因 |
| punishment_date | Date | 处分日期 |
| status | Integer | 1=生效中，0=已撤销（软删除） |
| created_at | DateTime | 创建时间 |

**audit_logs（审计日志表，只增不减）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer | 主键，自增 |
| operator_name | String(64) | 操作人姓名（冗余） |
| action_type | String(32) | LOGIN, QUERY_SINGLE, QUERY_BATCH, IMPORT, ADD, DELETE, RESTORE, BACKUP 等 |
| target | String(256) | 操作对象简述 |
| details | Text | 变更详情（JSON/Text） |
| timestamp | DateTime | 操作时间 |

### 4.2 数据关系

- 用户与审计日志：一对多。
- 名单与审计：通过 `action_type`、`target` 间接关联。

## 五、会话与认证

### 5.1 双重会话

- **客户端**：`st.session_state` 存登录状态、用户信息、最后活动时间；刷新会清空。
- **服务端**：`sessions.json` 存 token → 用户信息；token 经 URL 参数 `sid` 传递；刷新时 `app.py` 凭 `sid` 恢复登录态。

### 5.2 认证流程

```
用户输入工号/密码 → login.py BCrypt 校验 → 创建 session token
→ URL 写入 ?sid=token → st.rerun → app.py 识别 sid 恢复登录态 → 进入教师/管理员页
```

### 5.3 安全机制

- 密码：BCrypt 存储，不存明文。
- 登录限制：同一账号连续失败 5 次后 5 分钟冷却。
- 会话超时：30 分钟无操作退出，可配置提前提醒。
- 水印：登录后全局「姓名+工号」防截屏。
- 审计：登录、查询、导入、删除等关键操作写审计日志。

## 六、功能需求与实现

### 6.1 教师端（只读）

| 功能 | 需求要点 | 实现要点 |
|------|----------|----------|
| **单条查询** | 输入姓名或学号（支持回车）；仅查 `status=1`；学号完整显示；未命中提示「未查询到违规记录，该生信用良好」 | 按姓名或学号 `filter`，结果展示姓名、学号、专业、原因、处分日期 |
| **批量比对** | 上传 Excel（.xlsx/.xls）至少含「学号」列；学号清洗后与生效名单比对；无命中绿色提示，有命中红色警告+表格分页（每页 10 条）+ 可下载比对报告 Excel | `parse_batch_check_excel()` → 学号清洗 → 与 DB 生效名单比对（阶段二已按批 500 IN 查询）→ 分页展示与导出 |

### 6.2 管理员端

| 功能 | 需求要点 | 实现要点 |
|------|----------|----------|
| **数据仪表盘** | 名单总数、生效中、已撤销；按专业饼图；可选年份、近期变动 | Plotly 饼图/柱状图；阶段四仪表盘 `@st.cache_data(ttl=60)` |
| **名单管理** | 批量导入（姓名、学号、专业、原因、处分时间；学号清洗；已存在则更新）；单条新增（学号唯一）；生效列表筛选/排序/分页/软删除；已撤销列表与恢复；批量比对同教师端 | 阶段二生效/已撤销库内分页+筛选+排序；`parse_blacklist_excel()`；每 100 条 commit |
| **系统维护** | 审计日志最近 500 条、导出；手动下载 .db（SQLite）；MySQL/PostgreSQL 下提示 mysqldump/pg_dump；数据库恢复（仅 SQLite 上传覆盖） | `get_db_file_bytes()`（非 SQLite 抛提示）；恢复后 `st.cache_data.clear()` 并提示刷新/重启 |
| **用户管理** | 用户列表、新增用户、密码重置、启用/禁用（不可禁用当前登录账号） | 工号唯一、密码 BCrypt、角色 admin/teacher |

### 6.3 UI/UX 与约束

- **水印**：全局浅灰「姓名+工号」，防截屏。
- **交互**：耗时操作用 `st.spinner`；成功可 `st.balloons()`；错误用 `st.error` 等中文提示，不暴露 Traceback。
- **Excel**：.xlsx 用 openpyxl、.xls 用 xlrd；导入必需列见上；备份下载仅 SQLite 提供 `get_db_file_bytes()` 读 `database.db`。
- **环境**：`requirements.txt` 含 streamlit、sqlalchemy、bcrypt、pandas、openpyxl、plotly 等。

## 七、数据流与处理流程

### 7.1 批量导入

```
上传 Excel → utils.parse_blacklist_excel() → 学号标准化
→ 事务内逐条处理（每 100 条 commit，学号已存在则更新否则插入）
→ 记录审计 IMPORT → 显示统计
```

### 7.2 批量比对

```
教师上传 Excel → utils.parse_batch_check_excel() → 学号清洗
→ 查询 status=1 名单（学号按批 IN）→ 内存比对
→ 命中：表格+分页+下载报告；未命中：成功提示 → 记录审计 QUERY_BATCH
```

## 九、部署与运行

### 9.1 本地/单机

- **环境**：Python 3.9+；`pip install -r requirements.txt`。
- **首次**：`python init_db.py`（默认 admin/123456），**上线后务必修改默认密码**。
- **启动**：`python -m streamlit run app.py`；内网访问时 `--server.address=0.0.0.0 --server.port=8501`。
- **访问**：http://localhost:8501 或 http://服务器IP:8501。
- **数据位置**：项目根下 `database.db`、`sessions.json`、`backups/`、`logs/app.log`。

### 9.2 MySQL/PostgreSQL

- 设置 `DATABASE_URL`（如 `mysql+pymysql://...`、`postgresql+psycopg2://...`），安装 pymysql 或 psycopg2-binary。
- 空库：`python init_db.py`；从 SQLite 迁数据：使用项目迁移脚本（见 `docs/阶段一实施说明.md`）。
- 非 SQLite 下备份使用 mysqldump/pg_dump，应用内「下载 .db」会提示用上述工具。

### 9.3 多实例（阶段三）

- 多实例**必须**使用 MySQL 或 PostgreSQL，禁止 SQLite。
- 多进程 Streamlit（不同端口）+ Nginx 反向代理 + 会话粘滞（如 `hash $cookie_*`），WebSocket 需正确转发。示例：`deploy/nginx_example.conf`、`deploy/start_streamlit_workers.sh`。

### 9.4 部署示例（deploy/）

| 文件 | 说明 |
|------|------|
| nginx_example.conf | Nginx 多后端、会话粘滞、WebSocket |
| start_streamlit_workers.sh | 单机多进程启动（如 8501/8502/8503） |
| backup_example.sh | mysqldump/pg_dump 示例，需替换用户与库名 |

## 十、扩展性与约束

### 10.1 当前架构限制

- 单实例时 SQLite 写串行；数据规模建议 &lt; 10 万条；并发建议 &lt; 50（单实例）。
- 扩展后：支持 MySQL/PostgreSQL、库内分页与索引、多实例+会话粘滞（见阶段一～四）。

### 10.2 扩展方向（不删代码、不强制前后端分离）

- **阶段一**：DATABASE_URL、连接池、备份/下载兼容、迁移脚本与运维约定。
- **阶段二**：生效/已撤销库内分页+筛选+排序；审计导出分批/限条数；教师端批量比对学号分批 IN；Blacklist/AuditLog 索引。
- **阶段三**：Nginx+多 Streamlit 进程、会话粘滞、session_store 文件锁；多实例禁用 SQLite。
- **阶段四**：日志路径与打点、仪表盘短时缓存、备份示例脚本。

详见 `docs/阶段一实施说明.md`～`docs/阶段四实施说明.md`。

### 10.3 已实现优化（摘要）

- 名单：库内分页与筛选（阶段二）；教师端批量比对按批 500 IN（阶段二）；审计导出分批、限 50000 条（阶段二）。
- 索引：Blacklist (status, id_card)、(status, name)；AuditLog timestamp。
- 缓存：仪表盘 `@st.cache_data(ttl=60)`；恢复后 `st.cache_data.clear()`。
- 多实例：session_store 文件锁、多实例下 SQLite 检查与提示。

### 10.4 可选优化方向（未实现）

- 会话超时前提醒；当前用户修改密码；无命中时也展示/导出未命中名单；批量导入跳过行导出；审计导出与筛选一致；名单导出 Excel；初始化名单二次确认加强；SQLite WAL/写超时；仪表盘外短时缓存等。
