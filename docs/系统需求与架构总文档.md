## 一、系统概述与需求来源

**系统名称**：校园学术失信人员管理系统（Academic Dishonesty Management System）

**业务定位**（来源：`project_requirements.md`、`docs/千人规模评估与演进建议.md`）：
- 用于高校内部管理的 Web 系统，记录、公示和查询学术失信人员名单。
- 核心逻辑：Excel 导入名单库 → 自动清洗（学号等）→ 本地/外部数据库存储 → 前端公示与比对。
- 身份标识：名单以**学号**为唯一标识进行存储与比对（实现中可与旧库列名 id_card 兼容）。
- 部署环境：学校办公室局域网（Windows/Linux Server）。

**技术特点**（来源：`docs/系统架构总结.md`）：
- 轻量级部署：支持 SQLite 单文件数据库；可选 MySQL/PostgreSQL 以支撑多实例与千人规模。
- 简单易用：Streamlit 框架，快速构建 Web 界面。
- 安全可靠：BCrypt 密码加密、会话管理、审计日志、自动备份（SQLite 下）。
- 角色权限：教师（只读查询）和管理员（全功能管理）。

## 二、需求与功能（来源：project_requirements.md）

### 2.1 教师端（只读）

- **单条查询**：输入学生姓名或学号（支持回车提交）；仅查询 `status=1` 的生效记录；学号在界面与报告中完整显示。未命中时提示「未查询到违规记录，该生信用良好」。
- **批量比对**：上传 Excel（.xlsx/.xls），至少包含「学号」列；学号清洗后与数据库生效名单比对。无命中时绿色成功提示；有命中时红色警告、表格展示（分页，每页最多 10 条）、可下载比对结果报告 Excel（学号完整显示）。

### 2.2 管理员端

- **数据仪表盘**：名单总数、生效中数量、已撤销数量；按专业分布饼图（及阶段实现中的年份、近期变动等）。
- **名单管理**：批量导入（必需列：姓名、学号、专业、原因、处分时间；学号清洗；已存在则更新）；单条新增（学号唯一）；生效列表（筛选、排序、分页、软删除）；已撤销列表与恢复；批量比对同教师端。
- **系统维护**：审计日志（最近 500 条、导出）；手动全库备份下载（SQLite 下为 .db 文件，MySQL/PostgreSQL 下提示使用 mysqldump/pg_dump）；数据库恢复（仅 SQLite 下支持上传覆盖）。
- **用户管理**：用户列表、新增用户、密码重置、启用/禁用（不可禁用当前登录账号）。

### 2.3 UI/UX 与约束

- 水印防护：全局背景浅灰色水印「当前登录人姓名 + 工号」。
- 交互反馈：耗时操作使用 `st.spinner` 等；容错通过 `st.error` 等展示中文提示，不直接暴露 Traceback。
- Excel：.xlsx 用 openpyxl、.xls 用 xlrd；备份下载提供 `get_db_file_bytes()`，以二进制读取 `database.db`（仅 SQLite）。

## 三、技术栈与架构（来源：docs/系统架构总结.md、project_requirements.md）

| 层级       | 技术/工具        | 用途说明 |
|------------|------------------|----------|
| Web 框架   | Streamlit        | UI 构建、状态管理（st.session_state） |
| 数据库     | SQLite / MySQL / PostgreSQL | 单文件或外部库，由 `DATABASE_URL` 决定 |
| ORM        | SQLAlchemy       | 数据库交互、模型定义 |
| 数据处理   | Pandas、OpenPyxl/xlrd | Excel 读取与清洗 |
| 安全       | BCrypt           | 密码哈希加密与校验 |
| 可视化     | Plotly Express   | 管理员仪表盘图表 |
| 语言       | Python 3.9+      | 核心开发语言 |

核心业务流程：`Excel 导入名单 → 数据清洗（学号标准化）→ 数据库存储 → 前端公示与比对查询`。

架构分层（简要）：表现层（views：login、teacher_page、admin_page）→ 应用控制层（app.py：会话、路由、水印、超时、备份）→ 业务逻辑层（auth、utils、session_store）→ 数据访问层（database、models）→ 数据存储层（database.db / 外部库、sessions.json、backups/）。配置层：`core/config.py`。

## 四、数据模型（来源：docs/系统架构总结.md、project_requirements.md）

- **users**：id, username（工号唯一）, password_hash（BCrypt）, full_name, role（'admin'/'teacher'）, is_active。
- **blacklist**：id, name, student_id（表列名 id_card 兼容旧库）, major, reason, punishment_date, status（1=生效/0=已撤销）, created_at。
- **audit_logs**：id, operator_name, action_type（LOGIN/QUERY_SINGLE/QUERY_BATCH/IMPORT/ADD/DELETE/RESTORE/BACKUP 等）, target, details, timestamp。只增不减。

数据关系：用户与审计日志一对多；名单与审计通过 action_type/target 间接关联。

## 五、会话与认证（来源：docs/系统架构总结.md）

- 双重会话：客户端 `st.session_state` + 服务端 `sessions.json`，通过 URL 参数 `sid` 恢复刷新后的登录态。
- 安全机制：BCrypt 存储、登录失败限制（如 5 次后冷却）、会话超时（如 30 分钟）、水印、审计日志。

## 六、规模评估与演进（来源：docs/千人规模评估与演进建议.md）

- **结论**：不建议为「一千人同时使用、流畅不崩溃、上线服务器」而推倒重来；在现有架构上演进（换库 + 多实例 + 分页与导出优化 + 索引）即可达到目标。
- **推荐演进路线**（不删代码、不强制前后端分离）：
  - **阶段一**：数据库与连接 — SQLite 改为 MySQL/PostgreSQL（DATABASE_URL、连接池）、备份/下载兼容、迁移脚本与运维约定。
  - **阶段二**：查询与导出 — 生效/已撤销名单库内分页+筛选+排序，审计日志导出分批/限条数，教师端批量比对学号分批 IN，补充 Blacklist/AuditLog 索引。
  - **阶段三**：多实例部署与会话 — Nginx 反向代理、多 Streamlit 进程、会话粘滞（Cookie 或 IP）、session_store 多进程文件锁；多实例下必须使用 MySQL/PostgreSQL。
  - **阶段四**：运维与体验 — 日志路径与打点、仪表盘短时缓存、备份示例脚本。
  详见 `docs/阶段实施与审计.md`。

## 七、部署与运维（来源：docs/服务器上线部署指南.md、deploy/README.md、阶段一/三实施说明）

### 7.1 单机部署（办公室服务器）

- 环境：Python 3.9+，项目目录下 `python3 -m venv venv` 并 `pip install -r requirements.txt`。
- 数据目录：保证 `database.db`、`backups/`、`logs/` 可写；首次执行 `python init_db.py`（默认管理员 admin/123456），**上线后务必修改默认密码**。
- 监听：`python -m streamlit run app.py --server.address=0.0.0.0 --server.port=8501`，便于内网访问。
- 常驻：Linux 使用 systemd 配置 `WorkingDirectory`、`ExecStart`（同上命令）、`Restart=always`；Windows 可用计划任务或 NSSM 等。
- 防火墙：放行 8501（或反向代理 80/443）。
- 备份与运维：定期备份 database.db 与 backups/；查看 `logs/app.log`。非 SQLite 时使用 mysqldump/pg_dump（见阶段一生产备份建议）。

### 7.2 使用 MySQL/PostgreSQL

- 设置 `DATABASE_URL`（如 `mysql+pymysql://用户:密码@主机:3306/库名?charset=utf8mb4` 或 `postgresql+psycopg2://...`），安装 pymysql 或 psycopg2-binary。
- 空库时执行 `python init_db.py`；从现有 SQLite 迁数据时使用项目提供的迁移脚本（见 `docs/阶段实施与审计.md` 阶段一）。

### 7.3 多实例部署（阶段三）

- 多实例下**必须使用 MySQL 或 PostgreSQL**，禁止 SQLite。
- 启动多个 Streamlit 进程（不同端口），Nginx 反向代理 + 会话粘滞（如 `hash $cookie_*` 或 `ip_hash`），WebSocket 需正确转发。配置与脚本示例见 `deploy/nginx_example.conf`、`deploy/start_streamlit_workers.sh`。详见 `docs/阶段实施与审计.md`。

### 7.4 部署示例目录（deploy/）

| 文件 | 说明 |
|------|------|
| nginx_example.conf | Nginx 反向代理示例：多后端、会话粘滞、WebSocket。见 docs/阶段实施与审计.md。 |
| start_streamlit_workers.sh | 单机多进程启动脚本（默认端口 8501/8502/8503）。 |
| backup_example.sh | 生产备份示例（mysqldump/pg_dump），需替换用户、库名、路径。见 docs/阶段实施与审计.md。 |

## 八、可扩展性与未来优化（来源：docs/可扩展性改进指南.md、docs/功能优化建议.md）

以下仅归纳**已实现要点**与**文档中建议的未实现方向**，不编造新内容。

### 8.1 已实现的扩展与优化

- **名单变大**：生效/已撤销名单已改为库内分页+筛选+排序（阶段二）；教师端批量比对学号已按批 500 做 IN 查询（阶段二）；审计日志导出已分批、限制条数（阶段二）。
- **索引**：Blacklist 的 (status, id_card)、(status, name)，AuditLog 的 timestamp（阶段二）。见 `core/models.py`。
- **缓存**：仪表盘统计与图表已使用 `@st.cache_data(ttl=60)`（阶段四）；数据库恢复后调用 `st.cache_data.clear()`。
- **多实例**：session_store 多进程文件锁、Nginx 会话粘滞、多实例下 SQLite 检查与提示（阶段三）。
  详见 `docs/阶段实施与审计.md`。

### 8.2 文档中建议的未实现/可选方向（出处见下）

- **可扩展性改进指南**：SQLite 下可开 WAL、写超时；名单极大时对「姓名模糊查」可加 limit 或索引；仪表盘外如「操作人列表」等可短时缓存；配置项如 BATCH_CHECK_MAX_IDS、AUDIT_EXPORT_MAX_ROWS 等可按规模预留。
- **功能优化建议**：会话超时前提醒（如剩 5 分钟提示）；当前用户修改密码；教师端无命中时也展示/导出未命中名单；批量导入结果详情与跳过行导出；审计导出与筛选一致、生效/已撤销名单导出 Excel；初始化名单二次确认加强；大批量比对提示或分页策略；仪表盘「近期变动」按 updated_at（需先加字段）；管理员侧提供与教师端一致的单条查询/批量比对入口等。

实施顺序与细节见原文档（可扩展性改进指南中的「建议实施顺序」、功能优化建议中的「小结」优先级表）。若原文档已删除，可参考本总文档与各阶段实施说明。
