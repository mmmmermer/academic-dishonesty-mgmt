# 系统架构重构与优化全景方案

本文档整合了《项目现状与优化基础梳理》及《教师端大名单查询优化设计方案》的核心原则，形成了指导本系统后续开发的最高优先级路线与架构蓝图。在进行后续的代码开发时，必须严格遵守以下优化方案与安全设定。

---

## 一、系统底座架构重组 (清理 Ghost Refactoring 债务)

我们发现在历史开发中，存在新旧重构文件交织的乱象，尤其是工具模块与视图逻辑处于部分拆离、部分冗余的“半重构”状态。在添加新功能前亟需扫清这些技术债务：

### 1.1 核心巨石库 `core/utils.py` 的彻底消灭
- **文件拆离**：保留现有的 `core/excel_processor.py`、`core/audit_logger.py` 及 `core/student_id.py`。
- **安全池构建**：完全剥离 `safe_filename(raw)`、`remove_old_pdf(reason_field)` 及其配套的 `_PDF_DIR`，在防路径穿透的原则下构建全新的 `core/file_safe_guard.py` 模块。
- **全局引流**：系统级替换 `views/admin/*.py`、`views/login.py`、`views/components.py` 中散落的所有 `from core.utils import ...`，将其定向至上文中拆解好的各个独立组件上。最后**永久删除 `core/utils.py`** 以绝后患。

### 1.2 教师端逻辑“扇形解封”瘦身
- 废除 `views/teacher_page.py` 内超过 400 行的 UI 聚合代码。
- 将其降级为一个纯粹的 `UI 路由集线器`，所有实际的功能渲染（单条查询、批量比对、个人日志）均经由 `views/teacher/__init__.py` 引入外挂模块处理。

---

## 二、教师端“名单查询”模块复刻与安全加固设计

目前教师端仅拥有单管线的单条人员与 Excel 批扫能力。为提高审查泛用性，决定将基于拼音与全局检索的“名单大范围查阅”能力授权下放至教师端，但**要求对编辑与外发流做物理级的隔离阻断**。

### 2.1 路由结构层扩展
- 向瘦身后的 `views/teacher_page.py` 的 `TEACHER_NAV_OPTIONS` 追加 `› 名单查询` 功能项。
- 创建专有渲染子件 `views/teacher/list_query.py`，保持与其他模块的一致挂接方式。
- 引入 `teacher_effective` 作为该模块的缓存前缀隔离，杜绝与管理员端的 state 数据共振污染。

### 2.2 沉浸式只读大名单呈现逻辑
- **搜索复用**：完整沿用 `render_list_controls` 中的高级搜索挂件（“姓名/拼音/学号”及“单位分类框”）。
- **组件截断 (Key Feature)**：在下传调用 `render_blacklist_table` 表格构造器时，**禁止将任何 `selection_key` 注入其中**，从 UI 底层根源上让“复选框勾选器”被主动隐藏；禁止传授任何 `_show_edit_dialog` 或操作句柄。
- **导出阄割**：绝不唤起或渲染 `render_blacklist_export_button` 模块，断绝由教师端进行 Excel 全库下载的可能，维持该权限为仅有系统管理员独享。
- **结论文件审阅**：在表格展示列维护并允许显示认定结论文件的预览点击链接 (LinkColumn)，这可以增加普通教师审计失信凭证的直观依据。

### 2.3 功能红线约定
- **禁止包含提交门**：在 `views/teacher/list_query.py` 模块全域绝对不要牵动任何 `db.commit()`、`log_audit_action(AUDIT_ADD)` 或等效写行为。该模块定性为读请求最终出口（端点）。
- **反截屏防盗护盾常驻**：基于 `app.py` 中 `inject_watermark` 运行原理下的 DOM 水印将照常映射于新增查询列表背景图层。
- **保持现有翻页**：留存“上一页”、“下一页”以及基础“跳转”（`render_simple_pagination`）供教职人员在正常页面检索过渡用。
