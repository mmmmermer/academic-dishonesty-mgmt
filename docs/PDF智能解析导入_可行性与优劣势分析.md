# PDF 智能解析与批量导入功能：可行性方案与优劣势分析

## 1. 方案需求与业务痛点概述

### 1.1 业务目标
使管理员能够直接上传处分决定、违规通报等 PDF 文件，由系统自动抽取出**「姓名、学号、单位、违规原因、处分日期」**等核心要素，并将其批量导入到当前的学术道德黑名单数据库中，替代目前纯依赖标准 Excel 模板录入的操作。

### 1.2 核心业务挑战
*   **学号缺失（主键危机）**：公开发布或下发的通告中，为了保护隐私，通常会隐去完整的学号（如仅写名字，或学号打码 `U2021****`）。但在本系统中，学号是防止同名同姓误杀的**核心约束主键**。
*   **非结构化版式**：公文格式繁多，段落式的长文散文、“违规事实经过”与目标实体混杂，甚至可能遭遇带有鲜章的图片扫描件。单纯依赖正则表达式的提取准确率极差。

---

## 2. 拟定架构设计：三层人机协作（Human-in-the-Loop）架构

若需推进此功能，不可采用“解析即入库”的暴力做法，必须采用以下三层架构来守住数据质量底线：

1.  **版式解析层 (PDF Decoding & OCR)**：
    *   **数字型 PDF**：采用 `pdfplumber` 或 `PyMuPDF` 精准提取底层字符与潜藏表格。
    *   **扫描件 PDF**：接入机器视觉提取库（如 `PaddleOCR` 或大厂云端 OCR API）进行图片转写。
2.  **大语言模型语义抽取层 (LLM Extraction)**：
    *   摒弃死板的正则表达式。将全文片段投喂给轻量级的大模型（可云端 Kimi/DeepSeek/Qwen，或本地私有化部署模型）。
    *   利用 Prompt 强制 AI 返回标准的结构化目标：`[{"姓名": "...", "缺失学号标识": true, "判定原因": "..."}]`。
3.  **缓冲审核与人工校验池 (Human-in-the-Loop UI)**：
    *   **最关键的界面**：在 Streamlit 中运用 `st.data_editor`（全功能交互数据表）展示大模型的抽取结果。
    *   针对未提取到学号的行，系统自动标红，强制拦截入库行为；管理员需要根据显示的“姓名”和“单位”在工作台内**人工查验补充对应学号**。修正全表并点击验证后，才允许落库。

## 3. 方案优劣势深度剖析

### 🟢 核心优势 (Pros)

1.  **大幅降低初步整理的机械劳动**
    面对大量的历史长文红头文件，无需管理员一字一句去框选和复制粘贴。大模型具有出色的阅读和过滤能力，能在大段套话中瞬间提纯出被通报人的姓名和违规事实。
2.  **提取的鲁棒性极高**
    相比于代码写死的“模板截取”，AI 的语义提取抗干扰性极强。不管文件里是怎么叙述（如“经调查证实计算机学院张三...” 或者 “现对张三（计算机学院）...”），大模型均能统一格式化输出，不再受限于文件制作者的排版风格。
3.  **兼顾体验与数据绝对安全**
    “人机协作池”的设计是该方案的绝对亮点。它既享受了 AI 带来的速度红利，又因为拦截了最后一步入库权，彻底消除了（大语言模型固有的幻觉/张冠李戴）和（缺失主键带来的脏数据）风险。

### 🔴 劣势与实施成本 (Cons & Costs)

1.  **开发与持生成本剧增**
    *   **如果是云端方案**：需要对接商用 API，存在长期的 Token 计费成本，且将涉密/敏感的红头文件发往公网外面的 API 可能违反高校网络保密规定。
    *   **如果是本地部署**：需要额外在内网服务器部署拥有独立显存（16GB~24GB VRAM）的 GPU 服务器来跑本地 LLM（大语言模型）和本地 OCR，使得当前几乎零运维成本的 SQLite + Python 单体架构变得非常臃肿。
2.  **解析的延迟与性能瓶颈**
    提取 PDF（尤其是带 OCR 识别）叠加让大语言模型进行推理解析，耗时是巨大的（单份 PDF 可能耗时 10 秒到 30 秒）。在用户体验上，需要额外设计异步队列或加载遮罩，无法像过去导入纯 Excel 那样达到“毫秒级成百上千行”的爽快感。
3.  **无法做到 100% 无人值守（人工仍是瓶颈）**
    因为“隐私保护隐去学号”是外部通报的硬伤规律，大模型也无法凭空捏造学号出来。即便 AI 找出了 100 个作弊者，管理员最终还是必须在缓冲池界面，逐一去搜素并**手工填进那 100 个人的学号**，这个最耗费校验精力的最后一步是无法依靠技术去规避的。

---

## 4. 结论与后续建议

**实施建议：** 此为锦上添花而非雪中送炭的功能。
1.  **如果不做**：维持现有的 Excel 智能解析引擎，各学院上报时直接要求填入包含学号的标准化 Excel 是目前**成本最低、且数据质量最高**的管理方式。
2.  **如果要做**：强烈建议优先采用“混合缓冲池”配合云端轻量级 API 的模式进行小范围试错。务必在正式立项前，确认内网合规性以及获取真实样本 PDF 测试“无学号比例”有多大。如果超 80% 的通报都没有附带完整学号，不建议斥巨资开发自动化 PDF 提取系统。
