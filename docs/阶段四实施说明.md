# 阶段四实施说明：运维与体验

目标：规范**备份约定与示例**、**日志格式与关键路径打点**、**仪表盘短时缓存**，提升运维可观测性与高并发下体验。  
每步说明：**做什么 → 怎么做 → 作用与优化点**。

---

## 步骤 1：日志常量与统一路径（core/config.py + app.py）

### 做什么

- 将日志目录名、日志文件名等集中到 `core.config`，避免各模块写死 `"logs"`、`"app.log"`；主入口 `app.py` 使用这些常量构建 `LOG_DIR` 与日志文件路径。

### 怎么做

- 在 `core/config.py` 末尾增加：`LOG_SUBDIR = "logs"`、`LOG_FILENAME = "app.log"`（仅常量，不引入 logging 依赖）。
- 在 `app.py` 的 `_setup_logging` 中：从 config 导入 `LOG_SUBDIR`、`LOG_FILENAME`，`LOG_DIR = os.path.join(os.path.dirname(os.path.abspath(__file__)), LOG_SUBDIR)`，日志文件路径为 `os.path.join(LOG_DIR, LOG_FILENAME)`；若导入失败则回退为当前硬编码路径。

### 作用与优化点

- **作用**：运维或脚本需要轮转、重定向日志时，只需改 config 或环境；文档可明确写出「日志文件为 config.LOG_SUBDIR / LOG_FILENAME」。
- **优化点**：可读性提升，无虚构（常量均被 app 使用）。

---

## 步骤 2：关键路径打日志（views/login.py、views/admin_page.py）

### 做什么

- 在登录成功/失败、批量导入完成、名单删除/恢复、备份/恢复等关键操作处打一条 **info** 日志（不含密码、token 等敏感信息），便于排查与审计追溯。

### 怎么做

- **views/login.py**：在登录成功分支打 `logger.info("登录成功 user=%s", username)`；在「工号或密码错误」「账号停用」等分支打 `logger.info("登录失败 user=%s reason=...", username)`（reason 用简短固定文案，如 wrong_password / disabled）。
- **views/admin_page.py**：在批量导入成功、软删除/初始化名单、恢复为生效、备份下载/恢复等成功后打一条 info（可含操作类型与条数/文件名，不含用户密码）。使用 `logging.getLogger(__name__)`，与 app 的根 logger 同属一棵树，统一写入 `logs/app.log`。

### 作用与优化点

- **作用**：生产环境可通过 `logs/app.log` 快速定位谁在何时做了哪些操作，与审计日志表互补。
- **优化点**：仅打必要信息，不记录敏感数据；不虚构（每条日志对应真实代码路径）。

## 步骤 3：仪表盘短时缓存（views/admin_page.py）

### 做什么

- 仪表盘「名单总数/生效/已撤销」、专业分布与年份分布、近期变动等数据在短时间内不变，使用 Streamlit 的 `@st.cache_data(ttl=...)` 做短时缓存（如 60 秒），减轻数据库压力并加快重复打开仪表盘时的响应。

### 怎么做

- 将仪表盘所需数据拆成**无参或仅含可哈希参数的函数**，内部使用 `db_session()` 查询并返回可序列化结果（如 tuple、list、dict）。
  - 例：`get_dashboard_counts() -> (total, effective, revoked)`，`get_dashboard_major_counts()`、`get_dashboard_year_counts()`，`get_recent_blacklist_rows() -> list`。
- 用 `@st.cache_data(ttl=60)` 装饰上述函数（ttl 单位秒）；在 `_render_dashboard_metrics`、`_render_dashboard_charts`、`_render_dashboard_recent` 中改为调用这些 getter 而非直接使用传入的 `db`，因此这三个渲染函数改为无参，`_render_dashboard(db)` 改为 `_render_dashboard()` 并不再向子组件传 `db`。

### 作用与优化点

- **作用**：多人同时查看仪表盘或频繁切换 Tab 时，同一 60 秒内重复请求命中缓存，降低 DB 查询次数。
- **优化点**：逻辑清晰（数据获取与展示分离）；不虚构（getter 均被仪表盘渲染调用）。

## 步骤 4：备份示例脚本（deploy/）

### 做什么

- 提供**示例**脚本，展示生产环境如何用 mysqldump / pg_dump 做备份；不实现「应用内调用 DB 备份接口」，避免权限与依赖复杂化。

### 怎么做

- 在 `deploy/` 下新增 `backup_example.sh`（Linux/macOS）：内写注释与示例命令，如 `mysqldump -u用户 -p 库名 > backup_$(date +%Y%m%d).sql`、`pg_dump -U用户 库名 -f backup_$(date +%Y%m%d).sql`，并注明需替换用户、库名、路径；可选增加「仅结构」示例。不执行任何破坏性操作。
- 在 `deploy/README.md` 中增加对备份示例的说明，并指向阶段一/阶段四文档中的「生产备份建议」。

### 作用与优化点

- **作用**：运维可直接参考或复制脚本，减少误操作；与阶段一「生产备份建议」一致。
- **优化点**：不虚构（脚本仅示例命令，不依赖未实现的接口）。

---

## 阶段四完成后的状态

- 日志路径与文件名由 config 统一配置，主入口按此写 `logs/app.log`。
- 登录与管理员关键操作有 info 级别打点，便于排查与审计。
- 仪表盘数据 60 秒短时缓存，减轻 DB 压力。
- 提供备份示例脚本与文档衔接，生产备份有据可依。

---

## 运行前检查清单（阶段四）

1. **阶段一～三**：已完成数据库、分页/导出、多实例（若需）改造。
2. **日志**：确认 `logs/` 目录可写，或通过 config 修改 `LOG_SUBDIR`/`LOG_FILENAME`。
3. **缓存**：仪表盘 ttl 默认 60 秒，可在代码中按需调整 `@st.cache_data(ttl=...)`。
4. **备份**：生产环境按 `deploy/backup_example.sh` 或阶段一文档约定执行 mysqldump/pg_dump 或云快照。

---

上述步骤均不删除现有业务代码、不引入虚构接口；仅增加常量、日志、缓存与示例脚本，保持高可读性与可维护性。
