# 教师端“名单查询”模块复刻与优化设计方案

针对“将管理员端名单查询功能复刻优化到教师端”的需求，本方案旨在提升教师端的检索能力，同时通过物理隔离与逻辑截断，严控数据安全红线，确保教师仅拥有**只读查阅**权限。

## 核心设计决策 (Core Decisions)

根据您的最新指示，本方案已锁定以下关键细节：
- **不允许数据导出**：教师端将彻底移除所有 Excel 下载按钮，防止大名单数据离线扩散。
- **排序保持一致**：默认显示顺序与管理员端一致（学号字典序），维持多端操作直觉统一。
- **功能完全复刻**：保留高级“单位精准筛选”与“拼音智能检索”能力。
- **权限物理阉割**：在代码层面不引入任何编辑 Dialog 或勾选 Cart 逻辑，确保无写操作入口。

## 实施路线图

---

### 1. 路由与入口加固 (views/teacher_page.py)
在教师主页面中嵌入新模块，作为侧边栏的常驻功能。

- **新增导航项**：在 `TEACHER_NAV_OPTIONS` 中追加 `› 名单查询`。
- **按需加载方案**：仅在选中该板块时，通过 `render_teacher_list_query(db)` 动态拉取。
- **状态前缀隔离**：使用 `teacher_effective` 作为该板块下所有子组件的 session_state 前缀，防止与管理员端混淆。

---

### 2. 沉浸式只读查询视图 (views/teacher/list_query.py [NEW])
这是本次优化的核心业务文件，主要执行组件的拼装与权限裁剪。

- **控制栏复用**：调用 `render_list_controls`，开启“姓名/拼音/学号”检索以及“单位分类弹出选择器”。
- **表格只读渲染**：调用 `render_blacklist_table` 时，**显式不传递 `selection_key` 参数**。
  > [!NOTE]
  > 该底层组件在无 key 时会自动隐藏复选框，教师将无法进行勾选或触发批量操作。
- **认定结论预览**：保留“PDF 公示文件”的 LinkColumn 链接，方便教师核查原始认定凭证。
- **底部拦截**：不渲染 `render_blacklist_export_button` 与暂存区入口。

---

### 3. 数据安全与红线检查
- **无编辑探针**：全文件不包含 `_show_edit_dialog` 的任何逻辑。
- **无写入路由**：教师权限下无法触及任何 `db.commit()` 或 `log_audit_action(AUDIT_ADD)` 的业务代码块。
- **水印注入**：继承主程序的姓名+工号全局动态水印，防止教师通过截屏泄露名单信息。

## 开放性问题与确认

> [!IMPORTANT]
> **关于翻页体验的微调**
> 为防止教师通过脚本高频翻页恶意爬取完整名单，是否需要对教师端的“直接跳页 (Go)”功能进行移除，仅保留“上一页/下一页”的基础翻页？（目前方案暂定为保留基础翻页 `render_simple_pagination` 以提升普通用户的日常查找效率）。

## 验证计划 (Verification Plan)
- **视觉验证**：登录教师账号，确认 UI 布局中不含有任何“新建”、“修改”、“勾选”或“导出”按钮。
- **功能验证**：通过“【全选】计算机学院”测试级联筛选是否能准确滤出该学院所有学术失信人员。
- **安全验证**：尝试在 URL 中构造修改参数，确认后端是否有针对教师角色的权限校验拦截（已在 app 顶层路由中预设）。
