# 功能优化建议

基于当前代码与需求文档的梳理，以下为可考虑的优化方向（按优先级与实现成本分类）。

---

## 一、体验与安全类（优先考虑）

### 1. 会话超时前提醒
- **现状**：无操作超过 `SESSION_TIMEOUT_MINUTES` 后直接退出，无提前提示。
- **建议**：在距离超时还剩约 5 分钟时，用 `st.warning` 或 toast 提示「即将超时，请及时保存或操作以保持登录」，并可在前端记录「最后活动时间」做倒计时或简单提醒。
- **价值**：减少用户正在操作时突然被登出、未保存即丢失的情况。

### 2. 当前用户修改密码
- **现状**：仅管理员可为他人重置密码，用户无法自行修改自己的密码。
- **建议**：在侧边栏或「个人记录」旁增加「修改密码」入口：输入旧密码、新密码，校验旧密码后更新并写审计日志。
- **价值**：符合常见安全策略，减少对管理员的依赖。

### 3. 教师端批量比对：无命中时也保留「未命中名单」展示与导出
- **现状**：当上传名单**无人命中**时，直接提示「名单内学生信用良好」并清空 session，不展示、不导出未命中名单。
- **建议**：无命中时也写入 `teacher_batch_upload_rows`，展示「未命中名单」表格（即整份上传名单），并允许「下载未命中名单 (Excel)」，格式与现有未命中导出一致（姓名、学号、专业、原因、处分时间、是否命中）。
- **价值**：方便教师留存「本次全部未命中」的完整名单，便于归档或二次核对。

---

## 二、数据与操作类

### 4. 批量导入结果详情
- **现状**：导入后提示「新增 X 条，更新 Y 条」，若有跳过行仅写在审计详情中。
- **建议**：导入完成后在页面上简要展示「成功 X 条、更新 Y 条、跳过 Z 行（学号为空或格式异常）」；若有跳过，可提供「下载跳过行列表 (Excel)」或至少列出学号/行号，便于用户修正 Excel 后再次导入。
- **价值**：用户能快速发现并处理异常行，减少反复试错。

### 5. 审计日志导出列与筛选
- **现状**：审计日志可筛选操作人、类型、日期，导出为 Excel。
- **建议**：  
  - 导出列与列表展示一致（操作人、类型、对象、详情、时间等），并保证时间格式统一。  
  - 可选：导出时仅导出**当前筛选结果**，而不是固定「最近 500 条」，避免与页面所见不一致。
- **价值**：导出的 Excel 与界面筛选结果一致，便于审计与汇报。

### 6. 名单管理：生效/已撤销名单导出
- **现状**：生效名单、已撤销名单仅支持在线分页查看与排序。
- **建议**：在各自 Tab 下增加「导出当前名单 (Excel)」按钮，列与导入格式一致（姓名、学号、专业、原因、处分时间），可选「导出当前页」或「导出全部（含当前筛选）」。
- **价值**：便于备份、报送或线下核对，与导入格式统一。

---

## 三、易用与容错类

### 7. 教师端单条查询：结果列名与导入格式统一
- **现状**：单条查询结果表为「姓名、学号、专业、原因、处分时间」。
- **建议**：确认与 `REQUIRED_EXCEL_COLUMNS` 及批量比对展示列完全一致（当前若已统一为「处分时间」则无需改）。
- **价值**：用户在不同入口看到的字段一致，减少困惑。

### 8. 批量比对 Excel：列名容错
- **现状**：`parse_batch_check_excel` 仅要求「学号」列，其他列可选。
- **建议**：若上传表头含「处分日期」而无「处分时间」，已在构建 `upload_rows` 时映射为「处分时间」；可再检查是否对「专业」「原因」等做大小写/全角半角容错（若需求中有多种 Excel 模板）。
- **价值**：不同部门提供的 Excel 列名略有差异时仍可正常展示与导出。

### 9. 初始化名单二次确认
- **现状**：已有「确认初始化」与取消，且为危险操作。
- **建议**：可要求用户输入固定确认语（如「确认初始化」）或勾选「我已知晓将清空生效名单」后再显示「确认初始化」按钮，与数据恢复的确认强度一致。
- **价值**：降低误触导致整表被清空的风险。

---

## 四、性能与扩展（可选）

### 10. 大批量比对性能
- **现状**：批量比对将上传学号列表与黑名单做 `in_` 查询，命中结果与未命中名单均在内存中构建。
- **建议**：当上传行数极大（如 >5000）时，可考虑分块查询或仅输出「命中/未命中」统计 + 导出文件，避免一次性渲染超大表格；或保持当前逻辑但在 UI 上提示「名单过大时仅支持导出，不展示完整表格」。
- **价值**：在极端数据量下保持响应稳定。

### 11. 仪表盘「近期名单变动」数据源
- **现状**：近期名单变动按 `created_at` 排序，仅反映创建时间。
- **建议**：若业务上更关心「最近有变更」的记录，可考虑在 Blacklist 上增加 `updated_at` 字段，并在导入/编辑/恢复时更新，仪表盘按 `updated_at` 展示；若无该需求可维持现状。
- **价值**：使「近期变动」更贴近业务理解。

### 12. 管理员也具备「单条查询 / 批量比对」
- **现状**：需求文档写「管理员拥有教师端所有功能」，需确认当前管理员页是否已提供与教师端一致的单条查询与批量比对入口（若已存在则仅需在导航上更明显）。
- **建议**：若尚未提供，可在管理员侧边栏增加「查询/比对」入口，复用教师端逻辑，仅权限不同。
- **价值**：满足需求描述，管理员自查或演示时无需切账号。

---

## 五、小结

| 优先级 | 建议项 | 预期收益 |
|--------|--------|----------|
| 高 | 会话超时前提醒 | 减少误登出与未保存 |
| 高 | 当前用户修改密码 | 安全与自助能力 |
| 高 | 无命中时也展示/导出未命中名单 | 名单归档与核对 |
| 中 | 批量导入结果详情与跳过行 | 可追溯、易排错 |
| 中 | 审计导出与筛选一致 | 审计与汇报一致 |
| 中 | 生效/已撤销名单导出 Excel | 备份与报送 |
| 低 | 初始化名单二次确认加强 | 防误操作 |
| 低 | 大批量比对提示或分页策略 | 极端数据量下的稳定性 |

以上建议在功能上均可选做、分步实施；实现时建议优先做「会话超时提醒」「当前用户改密」「无命中时未命中名单展示与导出」三项，再按需推进数据与操作类优化。
