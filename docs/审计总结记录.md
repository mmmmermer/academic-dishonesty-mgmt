## 五、功能块与稳定性检查结论（审计）

| 功能块 | 检查结论 |
|--------|----------|
| 登录 | 成功/失败/异常分支均有日志；username_stripped 在 except 前已赋值，无未定义风险；Session 在 finally 中 close，无泄漏 |
| 会话恢复（sid） | get_session(sid) 从文件读，多进程已用文件锁；粘滞下同一用户固定实例，逻辑自洽 |

| 仪表盘缓存 | getter 内部用 db_session() 独立会话，返回可序列化类型；ttl=60 与恢复后 cache_data.clear() 一致 |

| 审计日志 | 列表与总数分两次查询，导出分批且限制 50000，无 OOM 风险 |
| 多实例 | MULTI_INSTANCE 仅 1/true 触发检查；session_store 读锁/写锁分离；文档与代码一致 |

## 六、阶段二审计与修复记录（实施说明中的记录）

- **LIKE 筛选**：对姓名/学号/专业筛选输入做 %、_、\ 转义（_like_escape），使用 LIKE ... ESCAPE '\\'，避免用户输入导致匹配过宽或行为异常。
- **审计日志查询**：列表条数与展示用 500 条改为两次独立查询（先 count()，再 limit(500).all()），避免依赖同一 Query 对象复用。
- **死代码与幻觉**：移除阶段二后已无调用的内存版筛选/排序函数（_filter_effective_list、_sorted_effective_list、_sort_key_by_attr、_filter_revoked_list），避免虚构引用。

来源：`docs/阶段二实施说明.md` 末尾「阶段二审计与修复（记录）」。

## 七、阶段三审计与修复记录（实施说明中的记录）

- **MULTI_INSTANCE 判定**：仅当环境变量为 1 或 true（不区分大小写）时执行 SQLite 检查并打日志，避免 MULTI_INSTANCE=0 等被误判为启用。
- **session_store**：为 _save_sessions 的 OSError 静默处理补充注释；为 _load_sessions 的过期清理说明多进程下为 best-effort、last-write-wins。

来源：`docs/阶段三实施说明.md` 末尾「阶段三审计与修复（记录）」。

## 八、安全性简要结论（审计）

- 凭证：DATABASE_URL 未写入日志、未向前端展示；登录日志仅记录工号与 reason，不记录密码。
- 注入：名单筛选使用参数化 + LIKE 转义；学号等由校验与清洗后入库。
- XSS：水印内容经 html.escape 再输出。
- 恢复：仅 SQLite 下开放上传覆盖，且需二次确认；非 SQLite 仅提示用 mysqldump/pg_dump。

未发现新增中高危问题；上述已改项均不引入新的安全缺口。来源：`docs/阶段一二三四审计报告.md` 第五节。
