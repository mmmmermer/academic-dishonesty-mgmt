# 当前方案 vs 换 MySQL/PostgreSQL + 多实例部署：差距说明

下面从**数据库**和**部署**两方面，说明「现在」和「换库 + 多实例」之间的具体差距，以及要动哪些地方。

---

## 一、当前方案（现状）

| 维度 | 现状 |
|------|------|
| **数据库** | SQLite 单文件 `database.db`，和项目同目录 |
| **连接** | 单进程内一个引擎、多个 Session，`check_same_thread=False` 允许多线程读；写是串行的，同一时刻只有一个写操作 |
| **部署** | 单机一条命令：`python -m streamlit run app.py`，一个 Streamlit 进程服务所有用户 |
| **会话** | `st.session_state` 存在该进程内存里，不落盘、不跨进程 |
| **依赖** | `requirements.txt` 里只有 sqlalchemy，无需单独装数据库服务 |

---

## 二、换 MySQL/PostgreSQL 的差距（仅换库、仍单实例）

### 2.1 和 SQLite 的差异

| 维度 | SQLite（当前） | MySQL / PostgreSQL |
|------|----------------|---------------------|
| **形态** | 一个 `.db` 文件，随应用一起 | 独立数据库服务，需单独安装、配置、启动 |
| **并发写** | 同一时刻只能有一个写事务，其他写会等锁 | 支持多连接并发写，锁粒度更细 |
| **连接数** | 通常单进程内一个连接池即可 | 可配置连接池大小（如 5～20），多用户同时查/写不会抢同一把锁 |
| **数据量** | 单库到 GB 级仍可用，再大要考虑备份、恢复速度 | 更适合大数据量、高并发读写 |
| **备份/恢复** | 直接复制 `.db` 或用 `get_db_file_bytes()` 下载 | 用 mysqldump / pg_dump 等工具，或云厂商快照 |
| **SQL 方言** | SQLAlchemy 已抽象，但如 `AUTOINCREMENT`、日期函数等有细微差别 | 换库后需用对应驱动，绝大多数 SQL 不变，少数函数/类型要核对 |

### 2.2 代码/配置上要改的

- **环境**：安装 MySQL（如 `mysqlclient` 或 `pymysql`）或 PostgreSQL（`psycopg2-binary`）驱动，并安装、启动数据库服务。
- **配置**：`database.py` 里不再用 `sqlite:///...`，改为从环境变量读 URL，例如：
  - MySQL：`mysql+pymysql://用户:密码@主机:3306/库名?charset=utf8mb4`
  - PostgreSQL：`postgresql+psycopg2://用户:密码@主机:5432/库名`
- **建库与迁移**：在 MySQL/PostgreSQL 里新建库，用现有 `models.py` 建表（或导出 SQLite 数据再导入），做一次数据迁移和回归测试。
- **连接参数**：去掉 SQLite 的 `check_same_thread`；可加连接池大小、超时等（如 `pool_size=10`, `pool_pre_ping=True`）。
- **业务代码**：`views`、`auth`、`utils` 里**不用改**，因为都是通过 SQLAlchemy Session 访问，不直接写 SQL；只有极少数地方若用了 SQLite 特有写法才需改成通用写法或对应方言。

**小结**：换库主要是**换 DATABASE_URL + 驱动 + 建库迁移**，应用层几乎不动，差距主要在「并发写能力、连接数、可维护的数据规模」。

## 三、多实例部署的差距（多开几个 Streamlit 进程）

### 3.1 和单实例的差异

| 维度 | 单实例（当前） | 多实例 |
|------|----------------|--------|
| **进程数** | 1 个 `streamlit run app.py` | 同一台机或不同机器上跑多个 Streamlit 进程（多端口或同一入口分流） |
| **负载** | 所有用户都由这一个进程处理，CPU/内存集中在一处 | 请求被分散到多个进程/机器，单机压力变小，可水平扩展 |
| **会话** | 所有用户共用一个进程，session 都在本进程内存 | **每个进程各自一份内存**，同一用户的两次请求可能被不同实例处理，**session 不共享** |
| **入口** | 直接访问该进程端口（如 8501） | 前面多一层 Nginx/云负载均衡，把请求分到多个端口或多台机 |

### 3.2 「多实例」带来的主要问题：Session 不共享

- **现状**：用户登录后，`st.session_state`（如登录态、当前选中的 Tab）存在**当前**响应的那个进程里。
- **多实例**：下一次请求可能被负载均衡分到**另一个**进程，那个进程里没有这名用户的 session，用户会**看起来像没登录或状态丢失**。

所以「多实例」要真正可用，要么：

1. **会话粘滞（推荐、最简单）**：在 Nginx/负载均衡里按 cookie 或 IP 做 **session affinity**，让同一用户尽量总是落到同一实例，这样 session 仍在单进程内存，和现在行为一致。  
   - **和现在的差距**：只是多了一层反向代理和配置，应用代码不用改。

2. **会话共享（复杂）**：把 session 存到 Redis/数据库，所有实例都读同一份。需要改应用里「读/写 session」的地方，不用 `st.session_state` 而改走 Redis/DB。  
   - **和现在的差距**：要引入 Redis（或 DB），并改 session 读写逻辑，工作量明显更大。

### 3.3 多实例时代码/运维上要动的

- **应用代码**：若只做**会话粘滞**，**不用改**；若做会话共享，要改 session 的存取方式。
- **部署方式**：用 systemd/supervisor/docker 等起多个 `streamlit run app.py --server.port=8502`、`8503`… 或每台机一个进程。
- **Nginx 示例**（会话粘滞）：  
  `upstream streamlit { server 127.0.0.1:8501; server 127.0.0.1:8502; ... }`  
  并配置 `hash $cookie_connect.sid consistent;`（或类似）让同一 cookie 总到同一后端。
- **数据库**：多实例会**同时**连数据库，若仍是 SQLite，多进程写会锁库，所以**多实例一般配合 MySQL/PostgreSQL** 使用更合适。

**小结**：多实例和现在的差距主要是「多进程、多端口/多机 + 负载均衡」；若用会话粘滞，对用户和开发来说和「单实例」体验一致，只是扩展了容量。

---

## 四、组合对比一览

| 项目 | 当前（SQLite + 单实例） | 换库（MySQL/PostgreSQL + 单实例） | 换库 + 多实例（会话粘滞） |
|------|-------------------------|-----------------------------------|----------------------------|
| 数据库服务 | 无，单文件 | 需安装、配置、备份 | 同左 |
| 并发写 | 串行，易锁等待 | 多连接并发写 | 同左 |
| 应用代码改动 | — | 仅 DATABASE_URL + 驱动 + 迁移 | 同左，session 不改 |
| 部署形态 | 单进程 | 单进程 | 多进程 + Nginx/负载均衡 |
| Session | 单进程内存 | 同左 | 每进程内存，靠粘滞保持同一用户同实例 |
| 扩展方式 | 只能竖扩（升配） | 竖扩为主 | 可横扩（加实例） |

---

## 五、总结：和现在的「差距」是什么

1. **只换 MySQL/PostgreSQL（仍单实例）**  
   - **差距**：多了一个独立数据库服务、连接方式和备份方式变了；并发写能力、连接数、可维护数据量都更好。  
   - **改动**：配置（含环境变量）、依赖、建库与数据迁移；业务代码几乎不变。

2. **换库 + 多实例（且用会话粘滞）**  
   - **差距**：在「1」的基础上，多了一层「多进程 + 负载均衡」，请求被分摊，可水平加机器/进程。  
   - **改动**：在「1」之外，增加部署与 Nginx（或等价）配置；**不**做 session 共享的话，应用仍不用改 session 逻辑。

3. **多实例且要做会话共享**  
   - **差距**：Session 从「进程内存」变成「Redis/DB」，可随意扩实例且不依赖粘滞。  
   - **改动**：在「2」的基础上，引入 Redis（或 DB）并改所有 session 的读写，工作量最大。

对你当前项目而言：**先换 MySQL/PostgreSQL 即可明显提升并发与规模；教师再多时再加多实例 + 会话粘滞，就能在不改 session 的前提下获得和现在一致的体验，只是容量更大。**
