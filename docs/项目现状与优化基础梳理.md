# 华中科技大学学术道德监督管理系统 — 项目现状与后续优化基础梳理

根据项目当前实际的代码结构、历史文档以及配置，对系统的技术栈、模块职责、安全机制及演进方向进行了全量且客观的梳理，严格遵循“禁止虚构”。本报告旨在为后续的二次开发、性能优化和模块解耦提供指引。

> [!NOTE]
> 当前应用为基于 **Python + Streamlit + SQLAlchemy** 构建的 Web 应用系统，并非 Vue/Django 架构。下文完全基于 `e:\hustsystem` 工作空间内的实际代码内容进行梳理。

---

## 1. 系统定位与当前技术栈

- **业务定位**：用于记录、公示、比对和管理“学术失信人员”信息的内部监督系统。
- **核心框架**：Streamlit（用于快速构建数据型 Web UI）。
- **对象关系映射 (ORM)**：SQLAlchemy。
- **数据库**：默认 SQLite，外加上已全面支持可迁移至 MySQL/PostgreSQL。
- **密码安全**：BCrypt 加密验证算法。
- **数据处理依赖**：Pandas, OpenPyxl, Xlrd（处理批量 Excel 导入、分析和比对导出）。
- **拼音与模糊检索关联依赖**：PyPinyin（结合 `core/search.py` 实现的高级拼音容错搜索引擎）。

---

## 2. 核心架构与目录结构

系统严格遵循 **展现层（Views） -> 核心控制层（App.py） -> 业务逻辑与数据访问层（Core）** 的单向依赖分层架构。禁止任何跨层级的闭环或逆向依赖。

### 核心目录映射
- `app.py`: **系统总入口**。管理请求生命周期，包括初始化 Session、鉴权检查（URL sid与会话超时恢复机制）、路由分发至 views 层、全局异常拦截保护与防 XSS 背景水印持久化注入。
- `core/`: **核心无状态/状态逻辑库**（纯 Python 逻辑层，完全脱离渲染）。
  - `models.py`: 三大核心底层表 `User`, `Blacklist`, `AuditLog`。
  - `database.py`: DB 连接池管理，开启了 SQLite WAL 加速及针对 MySQL/PG 的连接池优化配置。
  - `utils.py`: 巨石化的工具模块（集中承载了学号正则清洗、Excel 报表生成及拆解、防路径穿越的文件静态存储调度、全局敏感审计节点触发录入）。
  - `search.py` & `search_config.py`: 封装了专门的拼音化模糊搜索及快速匹配引擎算法模块。
  - `session_store.py`: 自定义的服务端会话文件锁存储器（`sessions.json`），以兼容多进程/多标签页下 Streamlit 特有的会话穿透态。
  - `config.py`: 项目全局宏文件（容纳了大量系统常量字面量、中文交互反馈文案、阈值、列表单页显示阈值设定）。
  - `auth.py`: 集成了单向不可逆哈希算法检查。
- `views/`: **展现层层（流式渲染控制层）**。
  - `components.py`: **通用 UI 组件库**（表格模块化渲染、防抖动的分页器、状态及专业筛选器通用封装）。
  - `login.py`: 独立登录全流程视图入口（含有内置的登录重试冷却频控闭环）。
  - `teacher_page.py`: 教师端（核心：工号单条验真查询，多名单批量 Excel 上传高速比对匹配分析以及比对报告反向导出）。
  - `admin/`子级目录: 管理员多重操作子台面视图切分：
    - `management.py`（单条手动修改插入、大量批量 Excel 合规强入覆盖与撤销控制软删除操作）。
    - `list_query.py`（采用沉浸式的名单跨域检索引擎、名单数据的全量合规过滤与分发导出入口）。
    - `system.py`（面向 IT 专员视角的只读审计流水线查询导出、SQLite 手动热备落盘与紧急文件级别的复写恢复操作面板）。
    - `user_mgmt.py`（全生命周期的账号开停流控制）。
- `部署外围脚手架`: `.streamlit/config.toml` (定制 Web 服务主题)、`start_system.bat`/`stop_system.bat`。

## 3. 核心机制与业务边界（强制不退让红线）

后续开发重构或功能深化推进中，必须绝对秉持以下底层架构公约：

1. **绝对主次键唯一性判断标识**：黑名单 `Blacklist` 处理对象任何业务关联（单查、批扫和自动比对映射）务必以 **工号/学号 (student_id)** 作为终局判断标识。模糊姓名在系统中的比对仅作为防重或预筛选辅助过滤，不得单独直接成为裁判标准。
2. **审计强制捆绑机制 (Audit Logging)**：不设例外地对任何形式的数据写操作，包括登入（无论成功与否的越界试探操作）、查单、大规模查密批扫与黑名单直接修订等，均强制调用 `utils.py` 内部 `log_audit_action` 落表（附详细信息细节）并保证 `AuditLog` 永不开启 UPDATE/DELETE SQL 操作层门槛。
3. **软删除理念闭合策略**：撤销与撤消名单的取消执行效力动作表现为 `status = 0` 做标记的纯软删除形式（辅以撤销时间戳和审计跟进），彻底严禁 DELETE DDL，以便满足学术或校属监管溯源。
4. **底层物理追踪防范**：通过注入全屏 HTML + CSS (`app.py/_inject_watermark`)，形成带有关联当前操作人权限与姓名的极强倾斜干扰防挡图层以反抗一切明面截图泄密手段。
5. **外部恶意请求投毒防御**：凡关联附件 (处理决意实体证据，如 PDF 引入) 实操存储均受 `utils.py/safe_filename` 以极严名目过滤，预防任何以 `../` 指令展开的 Directory Traversal 脱管穿越攻击。

## 4. 当前已经落实落地的重大进展与优化

系统在经历了先前的功能重叠与卡顿瓶颈后，目前已有以下强稳性基础表现（请勿对此做负优化退化）：
- **深层次并发争夺预防与 IO 改善**：在应对单库大批上传比对与名单存续交互频繁冲突时，完全启用了单文件级的 `SQLite WAL (Write-Ahead Logging)` 写前追加记录层技术规避独占所引发的表锁全站死机死锁；同时实现了从轻量级走向全天候业务型的跨引擎 `MySQL/PGDB` 双支持池优化方案预埋。
- **内存惰性吞吐化改良**：早期使用全表一波流的列表输出如今已被 `components.py` 等底层封装好的库内 `.limit().offset()` 拉取方式顶替取代，不仅是黑名单，包括繁冗庞大的日志审记出口现在也是进行分页加真实切分片段，完美保障 UI 长开不断线的极限状态。
- **智能化中文检索引擎补全建设**：依靠 `core/search.py` 构建完成支持超大批量（针对大范围生僻姓氏组合以及简拼需求）下的拼音转化字段与前置索引层支持引擎搭建，大大消除了早期由于人工建表字音误差识别带来的漏洞。

## 5. 后手拓展方案储备及优化方向预测（后续任务锚定点）

为了方便接下来的维护拓展，本报告提出以下重构性思考和底层强化铺垫（实施需基于任务独立发布）：

### 5.1 代码堆叠臃肿处的深度解剖与切割（迫在眉睫的主优化线）
- **剥离巨石库模块 `utils.py`**：当下核心工具集 `core/utils.py` 体积过于庞大，承担了从学号正则验证切片、Excel Pandas 表混合重塑生成、纯文件系统防穿越读存控制到全局不可控的日志埋点全生命监控等等，严重违反单一原则职责链。下步应优先将其进行按域物理打散，创建成诸如 `core/excel_processor.py`、`core/audit_tracer.py`及`core/file_safe_guard.py` 等分离集。
- **教师控制端 `views/teacher_page.py` 扇形解封**：单文件内同时涵盖着查单呈现、并发数万规模扫库报表合成、历史追溯的页面流展示逻辑极易带来代码合流灾难，亟待跟随目前 admin 的标准展开包裹化切分（新建 `views/teacher/batch_verifier.py` 与 `single_search.py` 拆包收拢）。

### 5.2 大规模应用层环境向云原生横纵延伸过渡（Cloud-Native）
- **服务端持久状态游离处理 (Session Redisization)**：假如后续要求本体系架设至 `Nginx 集群跨多服务器节点实例分发策略负载层` 下平摊算力，眼下仅依靠纯内存在硬盘落锁实现的 `session_store.py` 方案将会捉襟见肘，后续优化应引入一个 `Cache Abstract Layer` 转储于 `Redis/MemCached` 并完全摘清代码中当前有关本地 json 的锁死机制。
- **分离式高容量实证承载（针对处理决意附件）**：一旦大规模实施应用铺开，由程序自行挂接存放的 PDF 决意书静态卷将会在安全或备份管理形成压力（备份脚本会越跑越慢）。这必须转为全权对外交接交由 S3/MinIO OSS 云对象云池配置挂载接口托管存放与防盗链调用。

### 5.3 框架内状态流转抽象规整层封装 
- 应用当中存在大量因使用 Streamlit 机制带来的直插式 `st.session_state["MAGIC_STRING"]` 使用。尽管已经在 `config.py` 内以首字母大写的魔法常量压制进行收敛，但在深度逻辑模块穿梭（尤其是比对模块进行批扫和复写回传 UI 时），仍很容易由于漏键引发的未声明对象出错。考虑未来引入一个轻量化基于 python 的 Pydantic 数据规范检查或统一强类型的专属 StateManager 服务类来过滤与承接 UI 和数据的穿透管理（实现代码级强控与 IDE 自动联想防偏）。
