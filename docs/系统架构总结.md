# 校园学术失信人员管理系统 - 整体架构总结

## 一、系统概述

**校园学术失信人员管理系统**是一个基于 Python + Streamlit 的 Web 应用，用于高校内部管理、公示和查询学术失信人员名单。系统采用**单文件 SQLite 数据库**，部署简单，适合学校办公室局域网环境。

### 核心业务流程
```
Excel 导入名单 → 数据清洗（学号标准化）→ SQLite 存储 → 前端公示与比对查询
```

### 技术特点
- **轻量级部署**：SQLite 单文件数据库，无需额外数据库服务
- **简单易用**：Streamlit 框架，快速构建 Web 界面
- **安全可靠**：BCrypt 密码加密、会话管理、审计日志、自动备份
- **角色权限**：教师（只读查询）和管理员（全功能管理）

### 3.2 核心模块说明

#### **1. 应用入口层 (`app.py`)**
- **职责**：应用启动、路由分发、全局状态管理
- **核心功能**：
  - 会话状态初始化（`_init_session_state()`）
  - 从 URL `sid` 参数恢复登录态（`_restore_session_from_sid()`）
  - 侧栏导航渲染（`_render_sidebar()`）
  - 全局水印注入（`_inject_watermark()`）
  - 会话超时检测（`_check_session_timeout()`）
  - 启动时自动备份（`_run_auto_backup_once()`）
  - 全局异常捕获与日志记录

#### **2. 视图层 (`views/`)**
- **`login.py`**：登录页面
  - 表单校验（工号、密码）
  - 登录失败次数限制与冷却机制
  - 登录审计日志记录
  - 会话创建与 URL `sid` 注入
  
- **`teacher_page.py`**：教师端功能
  - 单条查询（按姓名或学号）
  - 批量比对（上传 Excel，与生效名单比对）
  - 比对结果导出（Excel 报告）
  
- **`admin_page.py`**：管理员端功能
  - 数据仪表盘（统计、饼图）
  - 名单管理（批量导入、单条新增、软删除、恢复）
  - 系统维护（审计日志、手动备份、数据库恢复）
  - 用户管理（新增、密码重置、启用/禁用）

#### **3. 业务逻辑层**
- **`auth.py`**：密码认证
  - BCrypt 密码校验（`verify_password()`）
  - 密码长度与编码校验
  
- **`utils.py`**：数据处理工具
  - 学号清洗（`clean_student_id()`）：去空格、全角转半角
  - Excel 解析（`parse_blacklist_excel()`、`parse_batch_check_excel()`）
  - 数据库备份（`auto_backup()`、`get_db_file_bytes()`）
  
- **`session_store.py`**：服务端会话存储
  - 基于 JSON 文件的 token 存储（`sessions.json`）
  - 会话创建、查询、删除
  - 过期清理机制

#### **4. 数据访问层**
- **`database.py`**：数据库连接管理
  - SQLAlchemy 引擎配置
  - 会话工厂（`SessionLocal`）
  - 上下文管理器（`db_session()`）
  
- **`models.py`**：数据模型定义
  - `User`：用户表（工号、密码哈希、姓名、角色、状态）
  - `Blacklist`：失信名单表（姓名、学号、专业、原因、处分日期、状态）
  - `AuditLog`：审计日志表（操作人、操作类型、目标、详情、时间戳）

#### **5. 配置层 (`config.py`)**
- 业务常量（审计操作类型、状态码、分页配置）
- UI 文案（标签、提示信息）
- 安全配置（会话超时、登录失败限制）
- Session 键名常量（避免魔法字符串）

## 四、数据模型设计

### 4.1 数据库表结构

#### **users（用户表）**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer | 主键，自增 |
| username | String(64) | 登录账号（工号），唯一 |
| password_hash | String(128) | BCrypt 加密后的密码 |
| full_name | String(64) | 真实姓名（用于水印显示） |
| role | String(16) | 角色：'admin' 或 'teacher' |
| is_active | Boolean | 账号状态（默认 True） |

#### **blacklist（失信名单表）**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer | 主键，自增 |
| name | String(64) | 姓名（索引） |
| student_id | String(32) | 学号（唯一，数据库列名为 id_card 兼容旧库） |
| major | String(128) | 所学专业 |
| reason | Text | 失信/违规原因 |
| punishment_date | Date | 处分日期 |
| status | Integer | 状态：1=生效中，0=已撤销（软删除） |
| created_at | DateTime | 创建时间 |

#### **audit_logs（审计日志表）**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | Integer | 主键，自增 |
| operator_name | String(64) | 操作人姓名（冗余存储） |
| action_type | String(32) | 操作类型（LOGIN, QUERY_SINGLE, QUERY_BATCH, IMPORT, ADD, DELETE, RESTORE, BACKUP） |
| target | String(256) | 操作对象简述 |
| details | Text | 变更详情（JSON/Text） |
| timestamp | DateTime | 操作时间 |

### 4.2 数据关系
- **用户与审计日志**：一对多（一个用户可产生多条审计日志）
- **名单与审计日志**：间接关联（通过操作类型和 target 字段记录）

## 五、会话与认证机制

### 5.1 双重会话管理

系统采用**双重会话机制**解决 Streamlit 刷新页面后 `st.session_state` 清空的问题：

1. **客户端会话**（`st.session_state`）
   - 存储当前登录状态、用户信息、最后活动时间
   - 页面刷新时会清空

2. **服务端会话**（`sessions.json`）
   - 存储 token → 用户信息的映射
   - Token 通过 URL 查询参数 `sid` 传递
   - 刷新页面时，`app.py` 根据 `sid` 恢复客户端会话

### 5.2 认证流程

```
用户输入工号/密码
    ↓
login.py 校验（BCrypt）
    ↓
创建服务端 session token
    ↓
写入 URL: ?sid=token
    ↓
刷新页面（st.rerun）
    ↓
app.py 检测 sid，恢复登录态
    ↓
进入对应角色页面（教师/管理员）
```

### 5.3 安全机制

- **密码加密**：BCrypt 哈希存储，不存储明文
- **登录失败限制**：同一账号连续失败 5 次后进入 5 分钟冷却期
- **会话超时**：30 分钟无操作自动退出，提前 5 分钟提醒
- **水印防护**：登录后全局显示「姓名+工号」水印，防止截屏泄露
- **审计日志**：所有关键操作（登录、查询、导入、删除等）记录审计日志

## 六、核心功能模块

### 6.1 教师端功能

#### **单条查询**
- 输入：学生姓名或学号（支持回车提交）
- 查询范围：仅 `status=1` 的生效记录
- 结果展示：姓名、学号、专业、原因、处分日期

#### **批量比对**
- 输入：上传 Excel（.xlsx / .xls），至少包含「学号」列
- 处理流程：
  1. 读取 Excel，提取学号列
  2. 学号清洗（去空格、全角转半角）
  3. 与数据库生效名单比对（基于学号）
  4. 生成比对结果报告
- 输出：
  - 命中记录：表格展示（分页，每页 10 条）
  - 未命中：绿色成功提示
  - 可下载 Excel 比对报告

### 6.2 管理员端功能

#### **数据仪表盘**
- 统计信息：名单总数、生效中数量、已撤销数量
- 可视化：按专业分布的饼图（Plotly Express）

#### **名单管理**
- **批量导入**：
  - 上传 Excel（必需列：姓名、学号、专业、原因、处分时间）
  - 自动数据清洗（学号标准化）
  - 学号已存在则更新记录
  - 批量提交（每 100 条 commit 一次，降低长事务）
  
- **单条操作**：
  - 手动新增（姓名、学号、专业、原因、处分日期）
  - 学号唯一性校验
  
- **列表管理**：
  - 生效名单列表（支持筛选：姓名、学号、专业）
  - 软删除（将 `status` 设为 0）
  - 已撤销名单列表
  - 恢复（将 `status` 设为 1）

#### **系统维护**
- **审计日志**：展示最近 500 条操作记录
- **手动备份**：下载当前 `database.db`（文件名带时间戳）
- **数据库恢复**：上传备份文件恢复数据库

#### **用户管理**
- 用户列表（工号、姓名、角色、状态）
- 新增用户（工号、密码、姓名、角色）
- 密码重置
- 启用/禁用账号（不可禁用当前登录账号）

## 七、数据流与处理流程

### 7.1 批量导入流程

```
上传 Excel 文件
    ↓
utils.parse_blacklist_excel() 解析
    ↓
数据清洗（学号标准化）
    ↓
数据库事务开始
    ↓
逐条处理（每 100 条 commit）
    ├─ 学号已存在 → 更新记录
    └─ 学号不存在 → 插入新记录
    ↓
事务提交
    ↓
记录审计日志（IMPORT）
    ↓
显示导入结果统计
```

### 7.2 批量比对流程

```
教师上传 Excel（含学号列）
    ↓
utils.parse_batch_check_excel() 解析
    ↓
学号清洗
    ↓
数据库查询（status=1 的生效记录）
    ↓
内存中比对（基于学号）
    ↓
生成比对结果
    ├─ 命中记录 → 表格展示 + 可下载报告
    └─ 未命中 → 成功提示
    ↓
记录审计日志（QUERY_BATCH）
```

- Python 3.9+
- 操作系统：Windows / Linux（适合学校办公室局域网）

### 10.1 架构优势

## 十一、扩展性考虑

### 11.1 当前架构限制
- **单实例部署**：SQLite 不支持多进程并发写入
- **数据规模**：适合中小规模数据（建议 < 10 万条记录）
- **并发能力**：适合中小规模用户（建议 < 50 并发）

### 11.2 扩展方向
详见 `docs/系统需求与架构总文档.md` 与各阶段实施说明：

1. **数据库升级**：SQLite → MySQL/PostgreSQL（支持更大数据量和并发）
2. **多实例部署**：负载均衡、会话粘滞（阶段三）
3. **性能优化**：数据库索引、库内分页、流式/分批导出（阶段二）
4. **功能扩展**：数据导入导出、报表生成、API 接口
