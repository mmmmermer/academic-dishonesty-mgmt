# 阶段三实施说明：多实例部署与会话粘滞

目标：通过**反向代理 + 多 Streamlit 进程 + 会话粘滞**实现水平扩展，支撑千人并发；保证同一用户请求固定到同一实例，避免 `st.session_state` 与 session 文件跨进程错乱。  
每步说明：**做什么 → 怎么做 → 作用与优化点**。

---

## 前置条件（必读）

- **数据库**：多实例下**必须使用 MySQL 或 PostgreSQL**，禁止使用 SQLite。多进程同时写 SQLite 会导致锁库与数据风险，阶段一已完成外部库与连接池支持。
- **会话**：应用使用 `st.session_state`（进程内存）+ `core/session_store.py`（sessions.json 持久化 token）。会话粘滞保证「同一用户固定到同一实例」；多实例共享同一 sessions.json 时已通过**文件锁**保证读写安全（见步骤 2）。

---

## 步骤 1：多进程启动与端口规划

### 做什么

- 在同一台机器或多台机器上启动**多个** Streamlit 进程，每个进程监听不同端口（或不同机则同端口亦可），由反向代理统一入口并做负载均衡与粘滞。

### 怎么做

- **单机多进程**：分别启动多个进程，绑定不同端口，例如：
  - 进程 1：`streamlit run app.py --server.port=8501 --server.address=0.0.0.0`
  - 进程 2：`streamlit run app.py --server.port=8502 --server.address=0.0.0.0`
  - 进程 3：`streamlit run app.py --server.port=8503 --server.address=0.0.0.0`
- 使用提供的 `deploy/start_streamlit_workers.sh`（Linux/macOS）或按需编写 Windows 批处理/计划任务，通过环境变量或命令行传入端口列表，循环启动。
- **多机**：每台机器运行一个或多个进程，在 Nginx/云负载均衡上游中配置多个 `server` 地址。

### 作用与优化点

- **作用**：多进程分摊请求，单机 CPU/内存瓶颈得以分散，整体 QPS 与并发能力提升。
- **优化点**：`--server.address=0.0.0.0` 允许本机以外访问，便于 Nginx 与负载均衡器转发。

---

## 步骤 2：session_store 多进程文件锁（代码改造）

### 做什么

- 多实例共享同一 `sessions.json` 时，`_load_sessions` / `_save_sessions` 可能并发读写，导致覆盖或损坏。为读写该文件增加**跨进程文件锁**，读用共享锁，写用排他锁。

### 怎么做

- 在 `core/session_store.py` 中：
  - 在**非 Windows** 平台使用 `fcntl.flock`：读文件时 `LOCK_SH`，写文件时 `LOCK_EX`，操作完成后 `LOCK_UN`。
  - 在 **Windows** 平台不加锁（单机多实例较少见，且 fcntl 不可用）；若需 Windows 多进程安全，可后续引入 `portalocker` 等跨平台锁库。
- 保持 `_load_sessions` / `_save_sessions` 的对外行为不变，仅在打开/写入文件时加锁、关闭前解锁。

### 作用与优化点

- **作用**：多实例共享同一 sessions 文件时，避免并发写导致数据丢失或 JSON 损坏。
- **优化点**：不改变现有 API；单实例或仅会话粘滞场景下，锁竞争可忽略。

---

## 步骤 3：Nginx 反向代理与会话粘滞

### 做什么

- 使用 Nginx（或云厂商负载均衡）作为统一入口，将请求转发到多个 Streamlit 后端；并配置**会话粘滞**，使同一用户的请求始终落到同一后端。

### 怎么做

- **基于 Cookie 的粘滞（推荐）**：Streamlit 会在浏览器设置 `_st_*` 等 Cookie；Nginx 使用 `hash $cookie_*` 或 `hash $cookie_*_st` 做一致性哈希，使同一 Cookie 的请求固定到同一 upstream 节点。
- 示例配置见本项目 `deploy/nginx_example.conf`：`upstream streamlit_backends` 列出多个 `server 127.0.0.1:8501` 等；`location /` 使用 `proxy_pass http://streamlit_backends`，并设置 `proxy_http_version 1.1`、`proxy_set_header Upgrade $http_upgrade`、`proxy_set_header Connection "upgrade"` 等，以支持 Streamlit 的 WebSocket。多进程启动脚本见 `deploy/start_streamlit_workers.sh`。
- **基于 IP 的粘滞**：若无 Cookie 或希望按 IP 固定，可使用 `ip_hash`，注意同一局域网多用户可能共享出口 IP，粘滞粒度较粗。

### 作用与优化点

- **作用**：同一浏览器会话的请求始终由同一 Streamlit 进程处理，`st.session_state` 与 session 恢复逻辑一致，避免「刷新后掉线」或状态错乱。
- **优化点**：WebSocket 与长连接正确转发，Streamlit 运行正常。

---

## 步骤 4：运行前检查与安全约定

### 做什么

- 明确多实例部署前的环境与安全要求：数据库类型、环境变量、日志与网络。

### 怎么做

- **数据库**：确认 `DATABASE_URL` 指向 MySQL 或 PostgreSQL，且所有实例使用**同一**连接串；禁止多实例下使用 SQLite。
- **环境变量**：各进程应使用相同的 `DATABASE_URL`、相同的应用配置（如 `core.config` 中的超时等）；如需区分实例可设置不同 `STREAMLIT_SERVER_PORT`，但不影响会话粘滞逻辑。若启用多实例部署，可设置 `MULTI_INSTANCE=1` 或 `MULTI_INSTANCE=true`，应用将在使用 SQLite 时打出一条警告日志，提醒改用 MySQL/PostgreSQL（仅上述二值会触发检查）。
- **sessions.json 路径**：多实例若共享同一部署目录，则共享同一 `sessions.json`，需保证该文件所在目录可写且已通过步骤 2 加锁；若多机部署且不共享文件系统，则需**会话粘滞到固定实例**，避免用户被调度到未持有其 session 的机器。
- **安全**：Nginx 与 Streamlit 之间建议走内网；对外只暴露 Nginx 的 80/443，不直接暴露 Streamlit 端口；HTTPS 在 Nginx 层配置，应用层可继续使用 HTTP（或通过 `X-Forwarded-Proto` 识别）。

### 作用与优化点

- **作用**：避免误用 SQLite、误配环境或暴露后端端口，保证多实例行为一致且可运维。
- **优化点**：集中约定便于交接与自动化部署。

---

## 阶段三完成后的状态

- 可在一台或多台机器上启动多个 Streamlit 进程，由 Nginx（或云 LB）做反向代理与负载均衡。
- 通过会话粘滞（Cookie 或 IP），同一用户请求固定到同一实例，`st.session_state` 与 sessions.json 行为正确。
- 多实例共享 sessions.json 时，读写由文件锁保护，无并发覆盖风险。
- 数据库为 MySQL/PostgreSQL，多进程共享同一库，无 SQLite 锁库问题。

---

## 运行前检查清单（阶段三）

1. **数据库**：已设置 `DATABASE_URL` 为 MySQL 或 PostgreSQL；未使用 SQLite。
2. **多进程**：已规划端口（如 8501、8502、8503），并已用脚本或进程管理器启动多个 `streamlit run app.py --server.port=... --server.address=0.0.0.0`。
3. **Nginx**：已安装并配置 `deploy/nginx_example.conf`（或等价配置），包含 upstream、会话粘滞与 WebSocket 相关 `proxy_set_header`。
4. **会话粘滞**：已启用 `hash $cookie_*` 或 `ip_hash`，且已验证同一用户多次请求命中同一后端（如通过日志或 Nginx 状态查看）。
5. **sessions.json**：若多实例共享同一部署目录，确认 `core/session_store.py` 已包含文件锁逻辑（步骤 2）；若多机不共享文件系统，确认粘滞到单机，避免跨机恢复 sid 失败。

---

## 安全与可读性约定

- **不暴露后端端口**：生产环境仅暴露 Nginx 80/443，Streamlit 端口仅监听内网或本地。
- **HTTPS**：在 Nginx 层配置 SSL，应用内可通过 `X-Forwarded-Proto` 识别协议。
- **日志**：各实例日志可统一写到 `logs/app.log`（同一机多进程会追加）或按端口/实例分文件，便于排查粘滞与错误。

---

## 阶段三审计与修复（记录）

- **MULTI_INSTANCE 判定**：仅当环境变量为 `1` 或 `true`（不区分大小写）时执行 SQLite 检查并打日志，避免 `MULTI_INSTANCE=0` 等被误判为启用。
- **session_store**：为 `_save_sessions` 的 OSError 静默处理补充注释（避免磁盘满等打断登录/登出）；为 `_load_sessions` 的过期清理说明多进程下为 best-effort、last-write-wins，便于后续维护理解。
