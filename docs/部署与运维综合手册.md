## <a id="A"></a>路线 A：高机密局域网部署与硬件指南 (Intranet Vault)

在极高保密与一般并发（几人~十几人同频使用）的特定场景下，系统的防御痛点不是“扛压防崩”，而是**“防内鬼抓包、防物理失窃、收缩网络攻击面”**。

### 1. 硬件落地方案
* **方案 1（性价比/便携流）**：零刻 SER6 Pro 迷你主机或 Intel NUC，连接内网交换机，置于保险文件柜中。性能极度溢出，支持长达十几年的数据生命周期。
* **方案 2（容灾冗余流，强烈推荐）**：群晖 DS723+ 等企业级 NAS。底层启用 RAID1 双盘镜像，一块盘物理损毁系统仍将无缝存活，保护所有原始 PDF 不丢。

### 2. 软件四重“铁桶防御”实施

1. **LUKS 全盘硬件层加密**：安装 Ubuntu 22.04 LTS (无桌面服务版)时，勾选 `LVM with LUKS`。服务器即使连盘带机被盗走，无开机主密钥仍为废钢。
2. **切断外部倾听，转内网**：不建议配置公网。Streamlit 后端启动时强制使用 `--server.address 127.0.0.1` ，并交由前端防火墙及 Nginx 在本机处理。
3. **Nginx 前置 HTTPS (防御网线监听)**：在哪怕全可信的办公室内网，所有数据交互也必须由 Nginx 挂载 SSL (自签名证书即可) 再传输，阻止 Wireshark 等嗅探软件反编译网络包。
4. **IP 白名单与软密码墙**：考虑到校园网乱串、大子网的通病，务必在 Nginx 的 `location /` 下设置 `allow 你的极小科室网段; deny all;`；或者要求网管静态绑定主机 MAC。无法协调的，必须增加 `auth_basic "Restricted Area"` 套第二重口令锁，防范学生扫描出后台。

## <a id="B"></a>路线 B：轻量级云服务器单机部署（公网 SQLite 版）

适用于 2核 4G 以下云主机（如阿里云 ECS / 轻量应用服务器），无需笨重的数据库组件，拿开即跑。

### 1. 系统与依赖基础

```bash
sudo apt update
sudo apt install -y python3 python3-venv python3-pip git nginx
git clone <本仓库地址> /opt/hustsystem
cd /opt/hustsystem
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
# 生成初始化表结构，会自带 admin/123456
python init_db.py 
```

### 2. Systemd 常驻运行
创建并修改 `/etc/systemd/system/hustsystem.service`：
```ini
[Unit]
Description=学术失信管理系统 (Streamlit)
After=network.target

[Service]
User=root   # 或者当前用户
WorkingDirectory=/opt/hustsystem
Environment="PATH=/opt/hustsystem/venv/bin"
# 启动时使用 headless
ExecStart=/opt/hustsystem/venv/bin/python -m streamlit run app.py --server.address=127.0.0.1 --server.port=8501 --server.headless=true
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload && sudo systemctl enable hustsystem && sudo systemctl start hustsystem
```

### 3. Nginx 代理映射至外网
创建 `/etc/nginx/sites-available/hustsystem`
```nginx
server {
    listen 80;
    server_name 你的域名或公网IP;
    client_max_body_size 50M;

    location / {
        proxy_pass http://127.0.0.1:8501;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        # WebSocket 支持是 Streamlit 的命脉，不能少以下三行
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 86400;
    }
}
```
配置生效即可由 80 端口进入 `sudo ln -s /etc/nginx/sites-available/hustsystem /etc/nginx/sites-enabled/ && sudo systemctl reload nginx`。最后再将系统防火墙以及云安全组放开 80 端口（并关闭先前测试用的 8501）。

*(注意：若这台服务器上原先已经存在别的旧 `streamlit` 进程占用 8501 端口，请首先寻址 `ss -tlnp | grep 8501` 并执行相关的 Kill 和清除操作)*

## <a id="C"></a>路线 C：高压高并发版 (MySQL/PostgreSQL) 迁移与上线操作

若需横跨分院进行多级管理，开启大量 `Streamlit` Worker 服务进程跑负载均衡，此时原生 SQLite 的本地写锁限制将成为瓶颈。

### 1. 挂载大型数据库实例

需要 MySQL 5.7+ (开启 utf8mb4 字符集) 或者 PostgreSQL 12+。
通过环境变量宣告 `DATABASE_URL` 强行切换为大型 RDS，接管系统的引擎分配。

Linux 系统中的注册表/环境变量修改：
```bash
# MySQL 版
export DATABASE_URL="mysql+pymysql://hustuser:你的密码@127.0.0.1:3306/hustsystem?charset=utf8mb4"

# PostgreSQL 版
export DATABASE_URL="postgresql+psycopg2://hustuser:你的密码@127.0.0.1:5432/hustsystem"
```

如果使用 `Systemd` 管理服务，请务必须并在 `.service` 配置中的 `.env` 位置（或直接写死）指向上述环境变量。 

### 2. 重建库与迁移

如果此节点先前是用 `路线 B` 实施积累了数据的 SQLite，请执行以下操作完成“大平移”：
1. 确保旧的 `database.db` 在根目录留存；
2. 为服务器装好并配置上述环境变量；
3. 执行：`python migrate_sqlite_to_external.py`，等待脚本无损完成导流。

**(切记：转入大 RDS 后，前端平台的“内建数据恢复功能”页面会提示停用，请转用原生的 `mysqldump` / `pg_dump` 按天备份！)**

## 🆘 常见问题集锦 FAQ

**Q1：为何访问 Nginx 给出的 80 网页一直白屏或报错"Please wait..."？**
> 答：90% 因为 Streamlit 在建立 WebSocket (WS) 通信时遇到了 Nginx 的掐断。请务必检查 Nginx 中是否保留了 `proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade";` 核心指令，并赋予足够的 `proxy_read_timeout`。

**Q2：上传 PDF 报 `Payload Too Large` 或者类似大小受限错？**
> 答：系统已通过 `config.py` 解开上传限制。若还阻断，一定是 Nginx 拦截。请检查是否已填写 `client_max_body_size 50M;` (依需调大)。

**Q3：我是多实例 Nginx （启动了多个 8501, 8502 端口转发负载均衡），为什么会话频繁登出或者上传错误？**
> 答：Streamlit 对 WebSocket 是有状态的关联绑定。如果是多实例负载池并发，Nginx Upstream 层必须加入 `ip_hash` 协议以确保当前浏览器每次连接到同一个 Worker PID，或者启用基于 Redis / Memcached 的外联存储中心。
