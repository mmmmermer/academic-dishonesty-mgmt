# 模糊搜索长期优化计划报告（评审稿）

文档状态：待评审  
适用系统：华中科技大学学术道德监督管理系统  
编制日期：2026-04-06  
执行原则：安全、封装、可维护、禁止虚构

## 1. 目标与边界

### 1.1 总目标

在不牺牲查询准确性和审计可追溯性的前提下，建设可长期运行的模糊搜索能力，支持未来数据规模增长，并适配：

- 单机部署（1 台主机）
- 小型桌面机部署（迷你主机/办公桌面机）

### 1.2 业务边界

- 教师端：默认“谨慎命中”，优先精确结果；纠错仅建议，不自动判定。
- 管理员端：可使用扩展召回，但保留可解释命中原因与审计日志。
- 批量比对：继续以学号精确比对为主，不与模糊逻辑混用。

### 1.3 非目标（本期不做）

- 不引入外部搜索引擎集群（Elasticsearch/Meilisearch）作为首期方案。
- 不启用高风险“错别字自动命中”。
- 不在未验证前宣称性能达标。

## 2. 强约束（必须满足）

### 2.1 安全约束

- 全部查询必须使用参数化 ORM，禁止字符串拼接 SQL。
- 所有查询行为记录审计日志（操作者、输入摘要、命中类型、时间）。
- 敏感数据最小暴露：日志中对姓名/学号做脱敏或摘要化存储。
- 数据库账号最小权限，禁止使用 root/超级用户直连业务。

### 2.2 封装约束

- 搜索解析、归一化、召回、排序统一收口在 `core/search.py`（必要时拆分为 `core/search_*` 子模块）。
- 页面层仅负责输入输出，不直接编写匹配规则。
- 规则权重和阈值配置化，不硬编码在页面。

### 2.3 可维护约束

- 采用标准迁移机制（建议 Alembic）管理数据库结构演进。
- 每个阶段都必须提供回归用例和验收记录。
- 文档与代码同步更新，杜绝“文档与现状不一致”。

### 2.4 禁止虚构约束

- 所有结论必须来自真实测试、真实日志、真实命令输出。
- 未执行测试不得写“已通过”。
- 性能结论必须附测试条件（数据规模、机器配置、并发方式、统计口径）。

## 3. 目标架构（长期）

### 3.1 数据层路线

- 生产与长期运行：PostgreSQL（单节点即可）。
- 开发/演示：可保留 SQLite，但仅作功能验证，不作为长期大数据运行方案。

### 3.2 搜索分层路线

按固定阶段召回，减少噪声并提升可解释性：

1. 精确层：学号精确、姓名精确、姓名去空格精确。
2. 可控模糊层：拼音全拼、拼音首字母、姓名前缀/子串。
3. 建议层：编辑距离/近似建议（仅当无高置信命中时展示建议，不自动命中）。

### 3.3 排序与解释

- 固定权重排序：`学号精确 > 姓名精确 > 规范化姓名 > 拼音全拼 > 拼音首字母 > 建议候选`
- 每条结果输出命中原因（`match_mode`）和分值（`match_score`）。
- 低于阈值不展示，避免无关结果污染。

## 4. 数据库与索引设计（PostgreSQL）

### 4.1 建议字段

在 `blacklist` 扩展搜索辅助字段：

- `name_norm`：姓名归一化（去空格、统一字符）。
- `name_pinyin_full`：姓名拼音全拼（小写、无音调、可无分隔）。
- `name_abbr`：姓名拼音首字母。

### 4.2 建议索引

- `(status, student_id)` 复合索引。
- `name_norm`、`name_pinyin_full`、`name_abbr` 单列索引。
- `status=1` 的部分索引（针对高频生效名单查询）。

说明：最终索引 SQL 在实施阶段按真实查询计划（`EXPLAIN ANALYZE`）定稿，不预先虚构性能收益。

## 5. 查询模块重构方案（封装）

### 5.1 模块职责切分

- `core/search_parse.py`：输入解析与多项分隔处理。
- `core/search_normalize.py`：姓名归一化、拼音计算。
- `core/search_match.py`：分层召回与匹配策略。
- `core/search_rank.py`：评分与排序。
- `core/search_config.py`：权重、阈值、开关配置。

注：若保持单文件也可，但必须按上述职责分段，保证可维护。

### 5.2 页面接入规则

- 教师页与管理员页统一调用同一搜索服务接口。
- 页面仅传入 `query_text`、`role_context`、`limit/page` 等参数。
- 页面不得自行实现拼音、模糊、阈值逻辑。

## 6. 部署适配方案（单机 + 小型桌面机）

### 6.1 部署剖面 A：单机标准版（推荐）

- 组成：`Streamlit + PostgreSQL` 同机部署。
- 适用：单部门长期运行，数据持续增长。
- 优点：部署简单、性能稳定、可维护性高。
- 资源建议：8GB 内存起步、SSD、定时数据库备份。

### 6.2 部署剖面 B：小型桌面机版（推荐）

- 组成：迷你主机（Windows/Linux）运行 `Streamlit + PostgreSQL`。
- 可选：内网反向代理（Nginx）做统一入口和访问控制。
- 适用：办公室长期在线、小规模并发访问。
- 运维重点：本地备份、离线副本、系统补丁、最小权限。

### 6.3 兼容策略

- 代码层保留 SQLite 兼容，但在大数据场景提示“非推荐”。
- PostgreSQL 路径作为长期默认，文档和脚本优先维护 PostgreSQL。

## 8. 测试与验收标准（禁止虚构）

### 8.1 功能验收

必须覆盖并记录：

- 姓名含空格：`张 伟` 命中 `张伟`
- 拼音全拼：`zhangwei`
- 首字母：`zw`
- 多项输入：换行/逗号/分号
- 管理员端避免过宽误命中
- 错字输入仅给建议（不自动命中）

### 8.2 安全验收

- SQL 注入测试样例不得影响系统数据。
- 未授权角色不得调用管理员扩展能力。
- 审计日志可追踪关键查询行为。

### 8.3 性能验收

- 在约定硬件与数据规模下测试 `P50/P95` 查询时延与错误率。
- 指标阈值由阶段 0 基线后确定并冻结，不预设虚构结论。

## 9. 风险与应对

- 风险：规则变复杂导致误命中上升。  
应对：分层召回 + 阈值控制 + 建议而非自动命中。

- 风险：字段回填对历史数据不一致。  
应对：回填前后校验、失败重试、可回滚迁移。

- 风险：桌面机资源波动影响稳定。  
应对：分页、索引、限流、定时维护与备份管理。

- 风险：长期保存全部数据库文件与备份会持续占用磁盘。  
应对：本机热备 + 外置冷备分层归档，开启容量监控与告警，不自动删除但可迁移到低成本存储介质。

## 12. 补充条款（本次新增）

### 12.1 变更治理条款

- 条款 G1：未经计划评审确认，不进入开发阶段。
- 条款 G2：每个阶段必须有可回滚点，回滚步骤写入文档并可演练。
- 条款 G3：禁止跨文件散落业务规则，搜索规则仅允许在搜索核心模块维护。

### 12.2 数据与安全条款

- 条款 S1：生产默认 PostgreSQL，SQLite 仅用于开发/演示。
- 条款 S2：审计日志保留期不少于 180 天，且保留查询命中类型与操作人。
- 条款 S3：数据库文件与备份采用“每日增量 + 每周全量 + 每月归档”，长期保存，不自动删除。
- 条款 S4：账号策略最小要求：最少 8 位密码、失败登录锁定策略、默认账号上线前必须改密。

### 12.3 单机/小型桌面机运行条款

- 条款 D1：最低硬件建议：4 核 CPU、8GB 内存、SSD。
- 条款 D2：必须启用分页查询和索引，不允许全表拉取后再做全量 Python 过滤。
- 条款 D3：单机部署需要本地自动备份；小型桌面机部署需额外提供离线备份副本。
- 条款 D4：系统资源保护：设置查询超时、单请求返回上限、输入长度上限。

### 12.4 验收与真实性条款（禁止虚构）

- 条款 T1：测试报告必须包含执行命令、测试时间、样本规模、测试结果四项证据。
- 条款 T2：未执行的测试项必须标记“未执行”，不得写“通过”。
- 条款 T3：性能结论必须附 `P50/P95` 指标和测试环境说明。
- 条款 T4：上线验收必须包含功能、安全、性能三类检查清单和签字记录。
