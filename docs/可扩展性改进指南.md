# 可扩展性改进指南

当**名单数量变大**或**教师用户数变多**时，可从以下方面逐步修改，在不大改架构的前提下提升稳定性与性能。

---

## 一、名单数量变大

### 1.1 生效/已撤销名单：全量加载 → 库内分页 + 筛选

**现状**（`admin_page.py` 等）：
- `effective_list = db.query(Blacklist).filter(Blacklist.status == 1).order_by(...).all()`
- 一次加载全部生效记录，再在内存里按姓名/学号/专业筛选、排序、分页。

**问题**：名单到几万条时，单次查询和内存占用都会明显上升。

**修改方向**（**已按阶段二实现**，见 `docs/阶段二实施说明.md`）：
- **筛选下推到 SQL**：姓名/学号/专业用 `filter(Blacklist.name.like(...))`（含转义）在数据库端完成。
- **分页在数据库完成**：`offset/limit` 只取当前页，排序用 `order_by(...)` 在 SQL 里做。
- **总数**：同条件的 `count()` 用于展示「共 N 条」和总页数。

实现位置：`_render_effective_list_section`、`_render_revoked_section` 内通过 `_effective_base_query` / `_revoked_base_query` + `_effective_order` / `_revoked_order` 及 `offset/limit` 完成。

### 1.2 教师端单条查询

**现状**：`teacher_page.py` 里按姓名或学号 `filter` 后 `.all()`，结果一般很少，影响不大。

**修改方向**：名单极大时，可对「姓名模糊查」加索引（见 2.1），或限制返回条数（如 `limit(100)`），避免单次返回过多。

### 1.3 教师端批量比对

**现状**（**已按阶段二实现**）：`student_ids` 按批 500 拆开，循环 `in_(batch)` 查询，结果合并。见 `views/teacher_page.py` 中 `_render_batch_check`。

### 1.4 审计日志导出

**现状**（**已按阶段二实现**）：导出按 `offset/limit` 分批取，最多 5 万条，并在界面提示。见 `_fetch_audit_logs_export_batched` 与 `_render_audit_log_display`。

### 1.5 仪表盘统计与图表

**现状**：`count()`、按专业/年份聚合用 `query(...).all()` 再在 Python 里统计。

**修改方向**：聚合尽量用 SQL 完成（`func.count`、`group_by`），减少 Python 端数据处理；数据量很大时可为仪表盘加短时缓存（见 2.3）。

---

## 二、教师用户数变多（并发与数据库）

### 2.1 数据库索引

**现状**（**部分已按阶段二实现**）：`Blacklist` 已有 `(status, id_card)`、`(status, name)` 及 `name` 单列索引；`AuditLog` 已有 `timestamp` 索引。见 `core/models.py` 与 `docs/阶段二实施说明.md`。

**可选的进一步优化**（在 `models.py` 或迁移脚本中加索引）：
- **Blacklist**：`(status, student_id)`、`(status, name)`、若常按处分日期筛可加 `(status, punishment_date)`，便于列表筛选和分页。
- **AuditLog**：`(timestamp DESC)`、若常按操作人/类型筛可加 `(operator_name)`、`(action_type)` 或组合索引，加速列表与导出条件查询。
- **User**：已有 `username` 唯一，一般够用；若按角色查可加 `(role)`。

索引后，名单/日志的 `filter + order_by + limit/offset` 会明显变快。

### 2.2 SQLite 并发与连接

**现状**：单文件 SQLite，`check_same_thread=False`，多线程可读；写仍串行，多管理员同时导入/修改会互相等锁。

**修改方向**：
- **开启 WAL 模式**：在 `database.py` 建连后执行 `PRAGMA journal_mode=WAL;`，读多写少场景下并发读更好，写锁等待时间更短。
- **写超时**：`create_engine(..., connect_args={"timeout": 15})`，避免写锁等太久无响应。
- **连接池**：SQLite 默认单连接，若用多进程部署（见 2.4），每个进程独立连接即可；不换库则一般不需额外连接池。

当**教师很多、且经常有多管理员同时操作**时，可考虑迁到 **MySQL 或 PostgreSQL**：改 `DATABASE_URL`，注意 SQL 方言（如 `LIMIT n OFFSET m`），并做一次数据迁移与回归测试。

### 2.3 缓存（可选）

**现状**（**仪表盘已按阶段四实现**）：`get_dashboard_counts`、`get_dashboard_major_counts`、`get_dashboard_year_counts`、`get_recent_blacklist_rows` 使用 `@st.cache_data(ttl=60)`；数据库恢复后会调用 `st.cache_data.clear()`。见 `views/admin_page.py` 与 `docs/阶段四实施说明.md`。

**可选的进一步优化**：
- 审计日志的「操作人列表」等也可短时缓存，减少重复 `distinct().all()`。

### 2.4 部署与多实例

**现状**：单进程 `streamlit run app.py`，多用户共用同一进程；`st.session_state` 在内存，无跨进程共享。

**修改方向**：
- **教师多、读多写少**：单机多用户通常没问题；若需水平扩展，可在前面加 Nginx 做负载均衡，多开几个 Streamlit  worker（如 `--server.maxUploadSize`、端口不同或同一入口分流）。注意：多实例下 session 不共享，用户可能被分到不同实例，一般可接受。
- **需要会话共享**：可考虑把 session 存到 Redis，并改 session 读写逻辑（Streamlit 默认不支持，需自行封装），工作量较大，一般名单/教师规模不必先做。

## 三、配置与常量

以下可在 `config.py` 中预先留好，便于按规模切换：

- **名单分页**：`LIST_PAGE_SIZE`、`LIST_PAGE_SIZE_OPTIONS` 已存在，名单变大后可把默认改为 20/50，并限制最大每页（如 200），避免单页过大。
- **批量比对**：可加 `BATCH_CHECK_MAX_IDS = 5000`，教师端单次上传学号数超过则提示分批。
- **审计导出**：可加 `AUDIT_EXPORT_MAX_ROWS = 50000`，导出超过则截断并提示。
- **数据库**：`DATABASE_URL` 从环境变量读取，便于同一套代码切到 MySQL/PostgreSQL（需安装对应驱动并改 URL）。

---

## 四、建议实施顺序

| 阶段       | 名单变大                     | 教师/并发变多                 |
|------------|------------------------------|-------------------------------|
| **先做**   | 生效/已撤销名单库内分页+筛选 | 索引（Blacklist、AuditLog）   |
| **其次**   | 审计日志导出流式或限条数     | SQLite WAL + 写超时           |
| **再考虑** | 批量比对学号分批 IN          | 仪表盘/列表短时缓存           |
| **规模更大** | 无                           | 迁 MySQL/PostgreSQL、多实例部署 |

按上述顺序改，可以在**不推翻现有架构**的前提下，支撑名单和教师数量上一个数量级；再往上则需考虑换库、会话与部署方案。
