# 一、阶段一实施说明：数据库与连接层改造

目标：支持 MySQL/PostgreSQL，增加连接池，为千人并发与多实例打基础。

**步骤 1 — 改造 core/database.py**  
从环境变量读取 `DATABASE_URL`，未设置时使用 SQLite。SQLite：`connect_args={"check_same_thread": False}`；否则：`pool_size=10, max_overflow=5, pool_pre_ping=True, pool_recycle=3600`。定义 `IS_SQLITE` 供 utils 等使用。

**步骤 2 — requirements.txt**  
增加 `pymysql`、`psycopg2-binary`，按需安装其一。

**步骤 3 — 兼容 core/utils.py**  
`get_db_file_bytes()`：若 `not IS_SQLITE` 则抛出异常，提示用 mysqldump/pg_dump。`auto_backup()`：若 `not IS_SQLITE` 则 return。

**步骤 4 — 迁移脚本**  
新增 `migrate_sqlite_to_external.py`：只读打开 SQLite，以当前 DATABASE_URL 为目标；目标上 create_all；按 User→Blacklist→AuditLog 批量写入（分批 commit）。先设好 DATABASE_URL 再执行；目标库为空也可直接 `python init_db.py`。

**步骤 5 — 文档与运维**  
文档中写明运行前准备、生产备份（MySQL: mysqldump；PostgreSQL: pg_dump）。

**运行前检查清单（MySQL/PostgreSQL）**  
1. 安装 pymysql 或 psycopg2-binary。2. 建空库，设置 DATABASE_URL。3. 空库执行 init_db.py；从 SQLite 迁数据执行 migrate_sqlite_to_external.py。4. 启动 streamlit。

**生产备份建议**  
MySQL：`mysqldump -u用户 -p 库名 > backup.sql`。PostgreSQL：`pg_dump -U用户 库名 -f backup.sql`。管理后台在非 SQLite 下提示使用上述工具。

# 二、阶段二实施说明：查询与导出优化

目标：名单库内分页+筛选+排序，审计导出分批/限条数，教师端批量比对学号分批 IN，补充索引。

**步骤 1 — 索引（core/models.py）**  
Blacklist：`Index("ix_blacklist_status_id_card", "status", "id_card")`、`Index("ix_blacklist_status_name", "status", "name")`。AuditLog：`Index("ix_audit_log_timestamp", "timestamp")`。

**步骤 2 — 生效名单（admin_page.py）**  
`_effective_base_query`（status==1 + 姓名/学号/专业 LIKE）、`_effective_order`；渲染处 base.count()、ordered.offset/limit 取一页；导出 `_fetch_effective_export_rows` 分批（每批 2000，最多 50000）。

**步骤 3 — 已撤销名单**  
`_revoked_base_query`、`_revoked_order`；同流程；导出 `_fetch_revoked_export_rows` 分批，最多 50000。

**步骤 4 — 审计日志导出**  
`_audit_log_export_query`；`_fetch_audit_logs` 返回 (logs, total)；`_fetch_audit_logs_export_batched` 分批，最多 50000；界面说明「共 N 条，最多导出 50000 条」。

**步骤 5 — 教师端批量比对（teacher_page.py）**  
学号按 BATCH_SIZE=500 分片，循环 `in_(batch)` 查询，结果 extend 合并。

**已有表建索引（表已存在时执行）**  
```sql
CREATE INDEX ix_blacklist_status_id_card ON blacklist (status, id_card);
CREATE INDEX ix_blacklist_status_name ON blacklist (status, name);
CREATE INDEX ix_audit_log_timestamp ON audit_logs (timestamp);
```
（MySQL/PostgreSQL/SQLite 同上。）

**阶段二审计与修复记录**  
LIKE 筛选：`_like_escape` 对 %、_、\ 转义，`LIKE ... ESCAPE '\\'`。审计列表：先 count() 再 limit(500).all()。移除死代码：_filter_effective_list、_sorted_effective_list、_sort_key_by_attr、_filter_revoked_list。

# 三、阶段三实施说明：多实例部署与会话粘滞

目标：多 Streamlit 进程 + 反向代理 + 会话粘滞。**多实例下必须 MySQL 或 PostgreSQL，禁止 SQLite。**

**步骤 1 — 多进程**  
多端口启动，如 8501/8502/8503；`--server.address=0.0.0.0`。可用 deploy/start_streamlit_workers.sh。

**步骤 2 — session_store 文件锁（session_store.py）**  
非 Windows：读 LOCK_SH，写 LOCK_EX，后 LOCK_UN。保持 _load_sessions/_save_sessions 对外行为不变。

**步骤 3 — Nginx**  
见 deploy/nginx_example.conf：upstream 多 server；proxy 设置 WebSocket（Upgrade、Connection "upgrade"）。粘滞：`hash $cookie_*` 或 ip_hash。

**步骤 4 — 运行前**  
DATABASE_URL 指向 MySQL/PostgreSQL；可设 MULTI_INSTANCE=1 或 true（SQLite 时打 warning）。sessions.json 共享则需文件锁；多机不共享则粘滞到单机。对外只暴露 Nginx 80/443。

**阶段三审计与修复记录**  
MULTI_INSTANCE 仅 1 或 true 时检查。session_store：_save_sessions 的 OSError 加注释；_load_sessions 过期清理说明多进程 best-effort。

# 四、阶段四实施说明：运维与体验

目标：日志常量、关键路径打日志、仪表盘短时缓存、备份示例脚本。

**步骤 1 — 日志常量**  
config：LOG_SUBDIR、LOG_FILENAME。app.py _setup_logging 用其构建路径。

**步骤 2 — 关键路径日志**  
login：成功/失败打 logger.info（不含密码）。admin_page：导入、删除/恢复、备份/恢复成功后打 info。

**步骤 3 — 仪表盘缓存**  
仪表盘数据拆成 getter，`@st.cache_data(ttl=60)`；_render_dashboard 无参并调用 getter。

**步骤 4 — 备份示例**  
deploy/backup_example.sh：mysqldump/pg_dump 示例命令，注明替换用户与库名。deploy/README.md 中说明并指向本文档。
